# Отчет по лабораторной работе:# Docker: сети, volumes, docker-compose

## 1. Проверка состояния сервисов (Healthchecks)
**Цель:** Убедиться в корректном запуске всего стека технологий (Frontend, Backend, Database) и прохождении проверок работоспособности.

* **Команда проверки:** `docker compose ps`
* **Результат:** Все сервисы (`db`, `backend`, `frontend`) находятся в статусе **Up** и имеют пометку **healthy**.
* **Вывод:** Стек успешно развернут. Использование инструкций `depends_on` с условием `service_healthy` позволило избежать ошибок при запуске, гарантируя, что бэкенд начал работу только после полной готовности базы данных.

## 2. Проверка связности компонентов (End-to-End)
**Цель:** Подтвердить корректность настройки сетевого взаимодействия между контейнерами и проксирования запросов через Nginx.

* **Команда проверки:** `curl localhost:8080/api/items`
* **Результат:** Получен ответ в формате JSON со списком объектов из базы данных:
  `[{"id": 1, "name": "apple"}, {"id": 2, "name": "banana"}, {"id": 3, "name": "cherry"}]`
* **Вывод:** Продемонстрирована работа полной цепочки данных: 
  1. Запрос от пользователя поступает на **Nginx** (порт 8080).
  2. Nginx проксирует его в контейнер **Flask** (внутренняя сеть проекта).
  3. Flask обращается к **PostgreSQL** за данными.
  4. Данные успешно возвращаются пользователю через всю цепочку.

## 3. Горизонтальное масштабирование (Scaling)
**Цель:** Продемонстрировать возможность быстрого увеличения вычислительных мощностей приложения.

* **Команда масштабирования:** `docker compose up -d --scale backend=3`
* **Результат:** Команда `docker compose ps` отображает три запущенных экземпляра сервиса `backend` (`backend_1`, `backend_2`, `backend_3`).
* **Вывод:** Инструментарий Docker Compose позволяет масштабировать отдельные компоненты системы одной командой без прерывания работы всего стека. Nginx автоматически (в рамках конфигурации upstream) может распределять нагрузку между этими экземплярами.
