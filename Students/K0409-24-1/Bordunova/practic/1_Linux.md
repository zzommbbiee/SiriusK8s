Запуском sudo `unshare --pid --fork --mount-proc /bin/bash` создан новый PID namespace, внутри которого команда `echo $$` показала `PID=1`, что подтверждает изоляцию процессов

Командой `sudo unshare --net /bin/bash` создан новый сетевой namespace, где `ip link show` отобразил только loopback-интерфейс, доказывая изоляцию сети

![подпись](/Students/K0409-24-1/Bordunova/images/photo11.png)

В блоке cgroups командой `sudo mkdir /sys/fs/cgroup/mytest` создана контрольная группа, и с помощью `echo "20000 100000" | sudo tee /sys/fs/cgroup/mytest/cpu.max` установлен лимит CPU 20%
Запущен процесс `stress-ng --cpu 2 --timeout 30s &`, его PID добавлен в cgroup командой `echo $! | sudo tee /sys/fs/cgroup/mytest/cgroup.procs`

Проверка через `cat /sys/fs/cgroup/mytest/cpu.stat` подтвердила, что лимит работает — процесс ограничен в потреблении CPU

![подпись](/Students/K0409-24-1/Bordunova/images/photo12.png)

В блоке chroot создан минимальный rootfs в /tmp/myroot, скопированы /bin/bash, /bin/ls и их зависимости с помощью `ldd` и цикла копирования. 
Командой `sudo chroot /tmp/myroot /bin/bash` выполнено погружение в изолированное окружение, где `ls /` отобразил только созданные каталоги bin, lib, lib64, proc, dev, что подтверждает изоляцию файловой системs

**Контрольный вопрос:** Почему после `exit` процессы хоста остались нетронутыми?

Потому что процессы, запущенные внутри нового PID namespace, имеют собственную иерархию идентификаторов. Для хоста они являются обычными дочерними процессами, и их завершение не влияет на остальные процессы системы. Каждый namespace изолирован — выход из него уничтожает только процессы внутри этого пространства

**Контрольный вопрос:** Что произойдёт если лимит памяти превысить? (OOM-killer)

Сработает OOM-killer (Out-Of-Memory killer) — механизм ядра, который принудительно завершит процесс(ы), превысившие лимит памяти, чтобы предотвратить крах всей системы. В логах (dmesg) появится запись об убитом процессе
