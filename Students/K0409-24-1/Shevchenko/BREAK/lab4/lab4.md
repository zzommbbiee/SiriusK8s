Так, ну в этой лабе я разбирался с неработающими бинарниками. Скрипт сгенерировал в директории /opt/break_lab пять файлов с разными поломками, и нужно было с помощью диагностических утилит разобраться в чем проблема и по возможности их починить.

1. Нет бита исполнения (mystery_no_exec)

Сначала я попробовал запустить файл, но получил ошибку команда не найдена (потому что bash не хотел его выполнять). Чекнул права через ls -l mystery_no_exec и увидел, что у него стоят дефолтные права только на чтение, а флага на выполнение нет. Просто накинул права через команду sudo chmod +x mystery_no_exec. После этого бинарник спокойно запустился и вывел текущую дату.

2. Кривой shebang (mystery_bad_interpreter)

При запуске система ругалась, что не может выполнить файл (No such file or directory), хотя файл вот он, лежит в папке. Глянул его тип командой file mystery_bad_interpreter. Оказалось, что это скрипт, но с битым интерпретатором. Я открыл его через nano и увидел в первой строке путь к какому-то несуществующему no_such_interpreter_break_lab. Чтобы пофиксить, этот путь надо просто заменить на валидный #!/bin/bash.

3. Усеченный ELF (mystery_truncated_elf)

При попытке запуска выдало ошибку не удаётся запустить бинарный файл. Тут я заюзал 3 способа. Сначала пробил через file, потом глянул заголовки через readelf -h mystery_truncated_elf. В выводе писалось: «Ошибка: Слишком много заголовков программы — файл не такой большой». Для верности прогнал запуск через strace ./mystery_truncated_elf и увидел системную ошибку ENOEXEC (Ошибка формата выполняемого файла). Понятно, что файл физически обрезан и восстановить его нельзя, так что тут чисто успешная диагностика.

4. Битый загрузчик (mystery_dyn)

Самое интересное. Файл не запускался с классической ошибкой No such file or directory. Чтобы найти причину, я чекнул зависимости библиотеки командой ldd mystery_dyn. Выяснилось, что путь к динамическому загрузчику ведет в никуда (был прописан /lib/THIS_INTERPRETER_DOES_NOT_EXIST). Чтобы это исправить, я использовал утилиту patchelf. Прописал правильный путь к линкеру командой sudo patchelf --set-interpreter /lib64/ld-linux-x86-64.so.2 mystery_dyn. После фикса всё заработало.

5. Текст под видом бинарника (not_a_binary)

Файл нормально выполнялся и выводил hello, но команда file not_a_binary показала, что это просто ASCII text (обычный текст), а не нормальный скрипт. Ядро его не понимало, поэтому файл выполнялся силами самого терминала. Чтобы это исправить, я открыл его в nano и добавил первой строкой правильный шебанг #!/bin/bash. После этого система стала четко видеть его как Bourne-Again shell script.
