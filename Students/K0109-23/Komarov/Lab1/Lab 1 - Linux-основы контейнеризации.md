#### Блок 1 — Изучение namespace-ов
Сначала были просмотрены namespace-ы текущего shell-процесса с помощью каталога /proc/ns.
Затем с помощью команды `lsns` был получен список namespace-ов, существующих в системе.

![[Pasted image 20260326182546.png]]
После этого был запущен новый процесс в отдельном PID namespace командой sudo unshare --pid --fork --mount-proc /bin/bash
![[Pasted image 20260326182913.png]]
Внутри нового пространства имён был проверен PID процесса

В результате внутри нового PID namespace процесс оболочки отображался как PID 1, что подтверждает изоляцию процессов.

Для создания отдельного сетевого пространства имён была использована команда sudo unshare --net /bin/bash
После запуска внутри нового окружения была выполнена команда ip link show

В выводе отображался только интерфейс lo (loopback), что подтверждает изоляцию сетевого стека и отсутствие доступа к сетевым интерфейсам хоста

#### Блок 3 — chroot
На следующем этапе был создан минимальный rootfs. В него были скопированы исполняемые файлы /bin/bash и /bin/ls, а также необходимые библиотеки, определённые через ldd /bin/bash. После этого был выполнен вход в новое корневое окружение
![[Pasted image 20260326184501.png]]
При создании chroot-окружения команда chroot выполнялась успешно, однако внутри окружения не запускалась команда ls. Причина заключалась в отсутствии необходимых разделяемых библиотек для /bin/ls, в частности libselinux.so.1. После копирования зависимостей команды ls с помощью ldd проблема была устранена. ### Ответы на контрольные вопросы

#### Почему после `exit` процессы хоста остались нетронутыми?
Shell внутри namespace видел только изолированные процессы своего окружения, а реальные процессы остались не тронуты
#### Что произойдёт если лимит памяти превысить? (OOM-killer)
OOM-killer завершит процесс, который потребляет слишком много памяти
