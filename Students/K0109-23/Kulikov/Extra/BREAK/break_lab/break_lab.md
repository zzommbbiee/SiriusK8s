# Linux Break Lab они же учебные поломки

скрипты создают контролируемые проблемы на Linux (лучше всего отдельная ВМ). После запуска нужно диагностировать и починить систему вручную, как в блоках с инфографики (nginx-лог, сеть, диск, бинарник, systemd)

## важно

- запуск только на тестовой ВМ, не на хосте с важными данными
- часть сценариев требует root (`sudo`)
- перед сетевым сценарием желательно иметь консоль провайдера ВМ или второй канал (serial), если SSH отвалится
- пути бэкапов: `/var/lib/linux_break_lab/`

## сценарии

| №   | скрипт                      | что ломает / создаёт                                                                     |
| --- | --------------------------- | ---------------------------------------------------------------------------------------- |
| 1   | `01_nginx_log_challenge.sh` | генерирует ~1M строк в `/var/log/nginx/access.log` (задача: топ-10 IP; опционально JSON) |
| 2   | `02_network_break.sh`       | портит маршрут по умолчанию и DNS                                                        |
| 3   | `03_disk_break.sh`          | заполняет root раздел большим файлом (+ опционально «удалённый, но открытый» файл)       |
| 4   | `04_binary_break.sh`        | несколько «незапускаемых» бинарей в `/opt/break_lab/`                                    |
| 5   | `05_systemd_break.sh`       | unit `script-server.service` с падающим процессом и лимитами ресурсов                    |

## запуск

```bash
cd PRACTIC/BREAK/break_lab
sudo bash 01_nginx_log_challenge.sh
# … остальные по одному, после починки предыдущего
```

## откат

- сеть: `sudo bash 99_restore_network.sh` (после `02_*`)
- диск: удалить файлы из вывода скрипта `03_*` и при необходимости убить фоновый `break_lab_orphan_holder`
- systemd: `sudo bash 99_restore_systemd.sh`
- nginx-лог и бинарники удалить вручную при необходимости (`rm` каталогов из вывода скриптов)

идеи починки: `ip`, `/etc/resolv.conf`, `nsswitch.conf`, `du`/`lsof`/`tune2fs`, `chmod`/`ldd`/`strace`, `journalctl`/`systemctl`

на этом все, удачи
