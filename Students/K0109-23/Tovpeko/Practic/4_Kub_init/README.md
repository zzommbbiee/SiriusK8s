## 1. Проверка состояния кластера (`kubectl get nodes`)

```bash
kubectl get nodes
# Пример вывода:
# NAME       STATUS   ROLES           AGE   VERSION
# minikube   Ready    control-plane   10m   v1.28.2
```

**Что я сделал:** Познакомился с командой `kubectl get nodes` для просмотра статуса узлов кластера.

**Зачем это нужно (я люблю Кирилла☺️):** Статус **`Ready`** означает, что нода исправна, kubelet работает и принимает поды. Если нода в статусе `NotReady`, планировщик (scheduler) не будет размещать на ней нагрузку. Это первый чек при диагностике кластера.

## 2. Проверка системных сервисов (`kubectl get pods -n kube-system`)

```bash
kubectl get pods -n kube-system
# Пример вывода:
# NAME                               READY   STATUS    RESTARTS   AGE
# coredns-787d4945fb-abcde           1/1     Running   0          10m
# etcd-minikube                      1/1     Running   0          10m
# kube-apiserver-minikube            1/1     Running   0          10m
# kube-proxy-xyz                     1/1     Running   0          10m
```

**Что я сделал:** Познакомился с пространством имён `kube-system` и проверил статус критических системных подов.

**Зачем это нужно:** В `kube-system` живут компоненты **Control Plane**:
*   **API Server** — «мозг» кластера, через него идут все запросы.
*   **etcd** — распределённая база данных, где хранится всё состояние кластера.
*   **CoreDNS** — сервисная сетевая адресация (превращает имена сервисов в IP).
*   **kube-proxy** — управление сетевыми правилами (iptables/IPVS).

Если хотя бы один из них не в статусе `Running`, кластер будет работать некорректно или не полностью.

## 3. Проверка рабочих приложений (`kubectl get pods`)

```bash
kubectl get pods
# Пример вывода:
# NAME           READY   STATUS    RESTARTS   AGE
# nginx          1/1     Running   0          5m
# my-webserver   1/1     Running   0          3m
```

**Что я сделал:** Познакомился с базовым управлением пользовательскими подами в дефолтном неймспейсе.

**Зачем это нужно:** Чтобы убедиться, что приложение успешно скачало образ, прошло проверку готовности (Readiness Probe) и начало обрабатывать трафик. Колонка `READY 1/1` означает, что главный контейнер в поде запустился и жив.

## 4. Демонстрация самовосстановления (`Self-healing`)

```bash
# 1. Находим имя контейнера внутри пода
kubectl get pod my-webserver -o jsonpath='{.spec.containers[0].name}'

# 2. "Убиваем" процесс внутри контейнера (эмуляция сбоя)
kubectl exec my-webserver -- kill 1
# или
kubectl delete pod my-webserver --force

# 3. Наблюдаем за рестартом
kubectl get pod my-webserver
# Пример вывода:
# NAME           READY   STATUS    RESTARTS   AGE
# my-webserver   1/1     Running   1          6m
```

**Что я сделал:** Познакомился с механизмом перезапуска контейнеров при сбоях, эмулировав падение процесса через `kill`.

**Зачем это нужно:** Это демонстрация ключевого принципа Kubernetes — **Self-healing (самолечение)**.
*   **Kubelet** на ноде постоянно мониторит состояние контейнеров.
*   Если процесс падает, kubelet автоматически перезапускает его (политика `restartPolicy: Always`).
*   Счётчик `RESTARTS` увеличивается, но приложение продолжает работать без ручного вмешательства оператора.

## Итог: архитектура кластера глазами админа

| Объект | Команда | Что проверяем | Почему это важно |
|--------|---------|---------------|------------------|
| **Нода** | `kubectl get nodes` | `STATUS=Ready` | Есть ли ресурсы для запуска подов? |
| **Control Plane** | `get pods -n kube-system` | Все `Running` | Работает ли «мозг» кластера (API, DNS, etcd)? |
| **Приложение** | `kubectl get pods` | `READY 1/1` | Запустилось ли бизнес-приложение? |
| **Resilience** | `RESTARTS > 0` после kill | Автоматический рестарт | Умеет ли система восстанавливаться сама? |
