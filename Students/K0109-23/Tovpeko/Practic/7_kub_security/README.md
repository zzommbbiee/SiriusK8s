## 1. Проверка прав на чтение (RBAC)

```bash
kubectl auth can-i list pods -n rbac-demo \
  --as=system:serviceaccount:rbac-demo:app-reader
# Вывод: yes
```

**Что я сделал:** Познакомился с проверкой прав сервисного аккаунта через `kubectl auth can-i`.

**Зачем это нужно:** Чтобы убедиться, что аккаунт `app-reader` имеет право `get/list` поды в неймспейсе `rbac-demo` согласно выданной `Role` и `RoleBinding`. Это основа принципа наименьших привилегий.

---

## 2. Проверка запрета на удаление (RBAC)

```bash
kubectl auth can-i delete pods -n rbac-demo \
  --as=system:serviceaccount:rbac-demo:app-reader
# Вывод: no
```

**Что я сделал:** Познакомился с проверкой отказа в деструктивных операциях.

**Зачем это нужно:** Чтобы гарантировать, что сервисный аккаунт не может удалять или изменять ресурсы за пределами своих полномочий. Это предотвращает случайные или злонамеренные изменения в кластере.

---

## 3. Блокировка трафика (NetworkPolicy)

```bash
kubectl exec frontend -- wget -q --timeout=2 database-svc
# Вывод: timeout (соединение отклонено)
```

**Что я сделал:** Познакомился с изоляцией трафика через NetworkPolicy, запретив доступ от пода `frontend` к сервису `database-svc`.

**Зачем это нужно:** Чтобы ограничить сетевое взаимодействие только разрешёнными путями. Даже если злоумышленник получит доступ к `frontend`, он не сможет напрямую подключиться к базе данных.

---

## 4. Разрешение легитимного трафика (NetworkPolicy)

```bash
kubectl exec backend -- wget -q --timeout=2 database-svc:5432
# Вывод: 200 OK / соединение установлено
```

**Что я сделал:** Познакомился с настройкой правил, разрешающих трафик от доверенного пода `backend`.

**Зачем это нужно:** Чтобы обеспечить работоспособность приложения: легитимные сервисы продолжают общаться, а все остальные соединения блокируются по умолчанию (deny-by-default).

---

## 5. Проверка сертификата (TLS)

```bash
openssl verify -CAfile ca.crt webapp.crt
# Вывод: webapp.crt: OK
```

**Что я сделал:** Познакомился с верификацией цепочки сертификатов через OpenSSL.

**Зачем это нужно:** Чтобы убедиться, что сертификат `webapp.crt` подписан доверенным корневым ЦС (`ca.crt`) и не был подменён. Это основа аутентификации сервера при установлении TLS-соединения.

---

## 6. Проверка HTTPS-соединения

```bash
curl --cacert ca.crt https://webapp.local
# Вывод: <!DOCTYPE html>... (ответ от nginx)
```

**Что я сделал:** Познакомился с установкой защищённого соединения к сервису через curl с проверкой корневого сертификата.

**Зачем это нужно:** Чтобы подтвердить, что:
*   Трафик шифруется по TLS.
*   Клиент проверяет валидность сертификата сервера.
*   Данные не передаются в открытом виде по сети.

---

## Итог: три уровня безопасности в Kubernetes

| Механизм | Что контролирует | Результат |
|----------|-----------------|-----------|
| **RBAC** | Кто и какие действия может выполнять с ресурсами | Запрет несанкционированных изменений |
| **NetworkPolicy** | Какие поды могут общаться друг с другом | Изоляция критических компонентов |
| **TLS** | Шифрование и аутентификация сетевого трафика | Защита данных от перехвата |
