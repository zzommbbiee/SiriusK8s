## 1. Масштабирование приложения (`kubectl get pods`)

```bash
kubectl get pods -l app=webapp
# Пример вывода:
# NAME                      READY   STATUS    RESTARTS   AGE
# webapp-7d89b8c5f4-abc12   1/1     Running   0          5m
# webapp-7d89b8c5f4-def34   1/1     Running   0          5m
# webapp-7d89b8c5f4-ghi56   1/1     Running   0          5m
```

**Что я сделал:** Познакомился с масштабированием приложения через `kubectl scale deployment/webapp --replicas=3`.

**Зачем это нужно:** Чтобы обеспечить **горизонтальное масштабирование**: несколько реплик пода обрабатывают больше трафика и повышают отказоустойчивость. Если один под упадёт, остальные продолжат обслуживать запросы. В production это основа High Availability.

## 2. История развёртываний (`kubectl rollout history`)

```bash
kubectl rollout history deployment/webapp
# Пример вывода:
# deployment.apps/webapp 
# REVISION  CHANGE-CAUSE
# 1         kubectl create -f webapp-v1.yaml
# 2         kubectl set image deployment/webapp webapp=webapp:v2
```

**Что я сделал:** Познакомился с командой `kubectl rollout history` для просмотра истории обновлений Deployment.

**Зачем это нужно:** Kubernetes хранит **ревизии** конфигурации Deployment. Это позволяет:
*   Отслеживать, какие изменения и когда вносились.
*   Быстро откатиться к рабочей версии при ошибке: `kubectl rollout undo deployment/webapp --to-revision=1`.
*   Реализовать стратегии безопасного деплоя (RollingUpdate, Blue-Green).

## 3. Проверка маршрутизации трафика (`curl`)

```bash
curl webapp.local
# Ответ: <html>...Welcome to WebApp...</html>

curl webapp.local/api
# Ответ: {"status":"ok","version":"2.1","endpoints":["/health","/api"]}
```

**Что я сделал:** Познакомился с проверкой работы приложения через разные эндпоинты, доступные через Ingress/Service.

**Зачем это нужно:** Чтобы убедиться, что:
*   **Service** корректно балансирует трафик между подами (через kube-proxy и iptables/IPVS).
*   **Ingress** (или CoreDNS) правильно резолвит имя `webapp.local` в ClusterIP сервиса.
*   Приложение отвечает на разные пути (`/` vs `/api`) — это проверка логики роутинга внутри самого приложения.

## 4. Типы сервисов: ClusterIP vs NodePort

### 🔹 ClusterIP (по умолчанию)

```yaml
spec:
  type: ClusterIP
  ports:
  - port: 80        # порт внутри кластера
    targetPort: 8080 # порт контейнера
```

| Характеристика | Описание |
|---------------|----------|
| **Доступность** | Только **изнутри кластера** (другие поды, сервисы) |
| **IP-адрес** | Виртуальный `ClusterIP` из диапазона `10.96.0.0/12` (не маршрутизируется наружу) |
| **Использование** | Внутренние микросервисы, БД, кэш, API-шлюзы |
| **Безопасность** | Высокая: сервис не виден извне без Ingress/PortForward |

> **Аналогия:** Внутренний телефон компании — набрать можно только с другого внутреннего номера.

---

### 🔹 NodePort

```yaml
spec:
  type: NodePort
  ports:
  - port: 80           # порт сервиса
    targetPort: 8080   # порт контейнера
    nodePort: 30080    # порт на ноде (30000–32767)
```

| Характеристика | Описание |
|---------------|----------|
| **Доступность** | **Извне кластера**: `<Node-IP>:<NodePort>` |
| **IP-адрес** | Открывает порт на **каждой ноде** кластера |
| **Использование** | Дев-окружения, быстрый доступ без Ingress, внешние интеграции |
| **Безопасность** | Ниже: порт открыт на всех нодах, нужен firewall/NetworkPolicy |

> **Аналогия:** Прямой городской номер сотрудника — можно позвонить извне, но нужно знать точный номер.

---

### 🔑 Сравнительная таблица

| Критерий | ClusterIP | NodePort |
|----------|-----------|----------|
| **Доступ извне** | ❌ Нет | ✅ Да (`<NodeIP>:3xxxx`) |
| **Порт** | Только внутри кластера | Фиксированный на ноде (30000–32767) |
| **Безопасность** | 🔒 Высокая | ⚠️ Требует дополнительной настройки |
| **Use Case** | Микросервисы, внутреннее общение | Dev/Staging, быстрый доступ, CI/CD |
| **В продакшене** | ✅ Через Ingress | ❌ Редко (лучше LoadBalancer/Ingress) |

## 🔑 Итог: как работает сетевая модель Kubernetes

```
┌─────────────────────────────────────┐
│  Внешний пользователь               │
│  (браузер, curl, другой сервис)     │
└─────────────┬───────────────────────┘
              │
    ┌─────────▼─────────┐
    │  Ingress /        │
    │  LoadBalancer     │  ← Точка входа извне
    └─────────┬─────────┘
              │
    ┌─────────▼─────────┐
    │  Service          │
    │  (ClusterIP)      │  ← Виртуальный IP + балансировка
    └─────────┬─────────┘
              │
    ┌─────────▼─────────┐
    │  Endpoints        │
    │  (список подов)   │  ← Динамический набор целевых подов
    └─────────┬─────────┘
              │
    ┌─────────▼─────────┐
    │  Pod (контейнер)  │
    │  порт 8080        │  ← Фактическое приложение
    └───────────────────┘
```

> **ClusterIP** — для внутреннего общения микросервисов.  
> **NodePort** — для «быстрого» доступа извне, но с оговорками по безопасности.  
> **Ingress** — правильный способ экспонировать приложения в production (единая точка входа, SSL, роутинг по доменам).
