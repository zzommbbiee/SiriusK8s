## 1. Три способа использования ConfigMap (`kubectl logs config-demo`)

```bash
kubectl logs config-demo
# Пример вывода:
# [INFO] DB_HOST=postgres.default.svc.cluster.local
# [INFO] DB_PORT=5432
# [INFO] Config file loaded: /etc/config/app.properties
# [INFO] APP_MODE=production
```

**Что я сделал:** Познакомился с тремя способами передачи конфигурации из ConfigMap в под:
1.  **Как переменные окружения** (`env.valueFrom.configMapKeyRef`) — отдельные ключи.
2.  **Как пакет переменных** (`envFrom.configMapRef`) — все ключи сразу.
3.  **Как файл** (`volumes.configMap` + `volumeMounts`) — монтирование в файловую систему.

**Зачем это нужно:** Чтобы реализовать принцип **12-Factor App**: конфигурация отделяется от кода. Это позволяет:
*   Менять настройки приложения без пересборки образа.
*   Использовать один образ для dev/stage/prod с разной конфигурацией.
*   Централизованно управлять настройками через объекты Kubernetes.

| Способ | Use Case | Пример |
|--------|----------|--------|
| `env` (по ключу) | Точечная настройка | `DB_HOST`, `API_KEY` |
| `envFrom` (все) | Массовый импорт | Загрузка всех настроек из одного источника |
| `volume` (файл) | Конфиг-файлы (`.conf`, `.yaml`, `.properties`) | `nginx.conf`, `appsettings.json` |

## 2. Работа с секретами (`kubectl get secret -o yaml`)

```bash
kubectl get secret db-credentials -o yaml
# Пример вывода:
apiVersion: v1
kind: Secret
type: Opaque
data:
  username: YWRtaW4=      # admin
  password: cGFzc3dvcmQxMjM=  # password123
```

**Что я сделал:** Познакомился с объектом `Secret` для хранения чувствительных данных и кодировкой base64.

**Зачем это нужно:** Чтобы **не хранить пароли, токены и ключи в открытом виде** в манифестах и коде:
*   **Base64** — это не шифрование, а кодирование (легко декодируется: `echo 'YWRtaW4=' | base64 -d`). Это защита от случайного просмотра, а не от злоумышленника.
*   В production секреты шифруются на уровне etcd (`EncryptionConfiguration`) или используются внешние хранилища (Vault, AWS Secrets Manager).
*   Секреты монтируются в под как переменные окружения или файлы в `tmpfs` (в памяти), что уменьшает риск утечки.

>  **Проверка:** `echo 'YWRtaW4=' | base64 -d` → `admin`

## 3. Проверка PersistentVolumeClaim (`kubectl get pvc`)

```bash
kubectl get pvc postgres-pvc
# Пример вывода:
# NAME            STATUS   VOLUME                CAPACITY   ACCESS MODES   STORAGECLASS   AGE
# postgres-pvc    Bound    pvc-a1b2c3d4e5f6      1Gi        RWO            standard       10m
```

**Что я сделал:** Познакомился с объектом `PersistentVolumeClaim` (PVC) и статусом `Bound`.

**Зачем это нужно:** 
*   **`Bound`** означает, что запрос на хранение (PVC) успешно привязан к реальному ресурсу (PV — PersistentVolume).
*   Это абстракция: разработчик запрашивает «1 ГиБ, доступ ReadWriteOnce», а админ кластера настраивает, где физически хранить данные (local-path, NFS, cloud disk).
*   Без PVC данные в контейнере живут только пока жив под — после удаления пода они исчезают.

## 4. Проверка сохранения данных после пересоздания пода

```bash
# 1. Записываем данные в БД до удаления
kubectl exec -it postgres-pod -- psql -U admin -c "INSERT INTO items (name) VALUES ('test-data');"

# 2. Удаляем под (эмуляция сбоя или деплоя)
kubectl delete pod postgres-pod

# 3. После перезапуска читаем данные
kubectl exec -it postgres-pod -- psql -U admin -c "SELECT * FROM items;"
# Пример вывода:
#  id |   name   
# ----+----------
#   1 | test-data
# (1 row)
```

**Что я сделал:** Познакомился с механизмом сохранения данных между жизненными циклами подов через PersistentVolume.

**Зачем это нужно:** Чтобы убедиться, что **данные не привязаны к конкретному контейнеру**:
*   Под — это «эфемерная» сущность, его можно удалить в любой момент.
*   PV/PVC — это «постоянное» хранилище, которое живёт независимо от пода.
*   При пересоздании пода с тем же `volumeClaim`, новый контейнер монтирует тот же том и видит старые данные.

Это фундамент для **Stateful-приложений** в Kubernetes: базы данных, очереди сообщений, файловые хранилища.

## Итог: управление состоянием и конфигурацией

| Объект | За что отвечает | Аналог в реальном мире |
|--------|----------------|------------------------|
| **ConfigMap** | Конфигурация приложения (не секретная) | Файл `settings.ini` или переменные `.env` |
| **Secret** | Чувствительные данные (пароли, ключи) | Сейф с ключами от серверов |
| **PersistentVolumeClaim** | Запрос на дисковое пространство | Заявка на аренду склада |
| **PersistentVolume** | Реальное хранилище (disk, NFS, cloud) | Сам склад с полками |

### Жизненный цикл данных в Kubernetes

```
┌─────────────────┐
│  Pod создаётся  │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│  PVC запрашивает│
│  хранилище      │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│  PV привязывается│
│  (status: Bound)│
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│  Том монтируется│
│  в /var/lib/post│
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│  Pod удаляется  │
│  (деплой, сбой) │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│  Данные на PV   │
│  СОХРАНЯЮТСЯ    │
└─────────────────┘
```

> **Stateless-приложения** (веб-сервисы) — можно удалять и создавать без последствий.  
> **Stateful-приложения** (БД, кэш) — требуют PVC, чтобы не терять данные.  
> **Конфигурация и секреты** — выносятся в ConfigMap/Secret, чтобы не пересобирать образы при изменении настроек.
