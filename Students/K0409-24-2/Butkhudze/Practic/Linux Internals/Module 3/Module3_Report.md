# Отчёт по лабораторной работе — Module 3


## Память, IOPS, inode и работа с tmux. Освобождение корневого раздела


### Цель работы


1. Научиться работать с **tmux** — создавать панели, переключаться между ними, включать синхронизацию.
2. Научиться мониторить систему в реальном времени (`free`, `vmstat`, `iostat`, `df -h`).
3. Освоить диагностику и исправление переполненного диска: поиск больших файлов, удаление «открытых, но удалённых» файлов через `lsof`, уменьшение резерва файловой системы.


---


## Краткое описание


Работа состоит из двух частей. В первой части я практиковался с tmux: открыл 4 панели, в каждой запустил утилиты мониторинга, а затем запустил `stress`, чтобы увидеть, как меняются показатели системы под нагрузкой. Во второй части я запустил скрипт‑поломку, который заполнил корневой раздел большим файлом и создал фоновый процесс, держащий удалённый файл, а моя задача была найти и устранить проблему без перезагрузки сервера.

## Часть A — Практика с tmux и мониторинг


### Установка необходимых пакетов


Перед началом я установил необходимые утилиты: tmux, stress и sysstat.


```bash

sudo dnf install tmux stress sysstat -y
```

```bash
asciinema rec --idle-time-limit 1 module3_tmux.cast
```

### Создание 4 панелей в tmux


Я запустил новую сессию tmux:


```bash
tmux new -s monitor
```


Затем разделил окно на 4 панели:


- `Ctrl+b "` — разделил окно горизонтально.
- `Ctrl+b %` — разделил верхнюю панель вертикально.
- `Ctrl+b ↓`, затем `Ctrl+b %` — разделил нижнюю панель вертикально.


В итоге у меня получилось 4 пустые панели, готовые для запуска команд мониторинга.


---


### Запуск мониторинга в четырёх панелях


В каждой панели я запустил свою утилиту:


| Панель                | Команда            | Что показывает                                      |
|-----------------------|--------------------|-----------------------------------------------------|
| 1 (верхняя левая)     | `free -h -s 1`     | Память (RAM и swap), обновление каждую секунду      |
| 2 (верхняя правая)    | `vmstat 1`         | Процессы, память, swap, IO                          |
| 3 (нижняя левая)      | `iostat -x 1`      | Нагрузка на диски (await, util и другие метрики)    |
| 4 (нижняя правая)     | `watch df -h`      | Свободное место на файловых системах                |


После запуска команд все 4 панели начали обновляться в реальном времени.


![Скриншот 3 — четыре панели tmux с запущенными командами free, vmstat, iostat и watch df -h](image3.png)


---


### Запуск стресс‑теста


Я создал новое окно в tmux (`Ctrl+b c`) и запустил нагрузку на систему:


```bash
stress --cpu 2 --vm 1 --vm-bytes 256M --timeout 30s
```


Затем вернулся в окно с мониторингом (`Ctrl+b 0`) и наблюдал изменения показателей:


- В `free -h -s` уменьшилось количество свободной памяти примерно на 256 МБ.
- В `vmstat` колонка `us` (user time) выросла до 50–80%.
- В `iostat` появилась дополнительная активность по CPU и диску.


![Скриншот 4 — работа stress: на показателях видно рост нагрузки на CPU и память](image4.png)


---


### Синхронизация панелей tmux


Чтобы протестировать синхронный ввод, я включил режим синхронизации панелей:


```text
Ctrl+b :setw synchronize-panes on
```


После включения я ввёл команду `echo "test"` — она одновременно выполнилась во всех четырёх панелях. Затем я выключил синхронизацию командой:


```text
Ctrl+b :setw synchronize-panes off
```


![Скриншот 5 — включённая синхронизация панелей, одна и та же команда выполняется во всех окнах](image5.png)


---


### Завершение работы с tmux и загрузка записи


Когда задания с мониторингом были выполнены, я:


```bash
Ctrl+b d          # отцепился (detach) от сессии tmux
exit              # остановил запись asciinema
asciinema upload module3_tmux.cast
```


После загрузки asciinema показал ссылку на запись.

## Часть Б — Освобождение корневого раздела (Практическое задание №3)


### Запуск скрипта‑поломки


Согласно лабораторному заданию я перешёл в нужную директорию и запустил скрипт, который создаёт проблему с заполнением диска:


```bash
cd PRACTIC/BREAK/break_lab
sudo bash 03_disk_break.sh
```


Скрипт создал большой файл и запустил фоновый процесс, держащий этот файл открытым, после чего вывел на экран путь и PID процесса.


![Скриншот 8 — вывод скрипта 03_disk_break.sh с информацией о созданном файле и PID процесса](image8.png)


---


### Первичная диагностика заполнения диска


Сначала я проверил заполненность корневого раздела:


```bash
df -h /
```


Из вывода было видно, что корневой раздел почти полностью забит (около 95–100%).


![Скриншот 9 — вывод df -h /, корневой раздел практически заполнен](image9.png)


Чтобы найти крупные файлы, я посмотрел размеры в рабочей директории скрипта:


```bash
sudo du -sh /var/lib/linux_break_lab/*
```


Там нашёлся большой файл `disk_fill.bin` размером примерно 4 ГБ.


![Скриншот 10 — du -sh показывает большой файл disk_fill.bin около 4 ГБ](image10.png)


---


### Удаление большого файла


Я удалил найденный файл и сразу проверил место:


```bash
sudo rm -f /var/lib/linux_break_lab/disk_fill.bin
df -h /
```


Свободное место увеличилось, но не на всю ожидаемую величину — это означало, что файл всё ещё открыт каким‑то процессом и его место не до конца освобождено.


![Скриншот 11 — df -h / после rm: место освободилось только частично](image11.png)


---


### Поиск «удалённого, но открытого» файла


Для поиска процессов, держащих удалённые файлы, я использовал:


```bash
sudo lsof | grep deleted
```


В выводе был виден процесс `break_lab_orphan_holder`, который держал удалённый файл `disk_fill.bin`.


![Скриншот 12 — вывод sudo lsof | grep deleted, видно процесс break_lab_orphan_holder и удалённый файл](image12.png)


---


### Завершение процесса‑держателя


Я завершил этот процесс по PID из специального файла:


```bash
sudo kill $(cat /var/lib/linux_break_lab/orphan_holder.pid)
# альтернативный способ:
# sudo pkill -f break_lab_orphan_holder
```


После этого убедился, что процесс больше не работает:


```bash
ps aux | grep break_lab_orphan_holder
```


![Скриншот 13 — процесс break_lab_orphan_holder отсутствует в списке процессов](image13.png)


---


### Контроль свободного места


После завершения процесса я снова проверил корневой раздел:


```bash
df -h /
```


Теперь свободное место увеличилось на полные 6 ГБ — диск вернулся к нормальному состоянию, и корневой раздел больше не переполнен.


![Скриншот 14 — df -h / после исправления, свободного места достаточно](image14.png)


---


### (Опционально) Уменьшение резерва файловой системы


Дополнительно я проверил и при необходимости уменьшил процент зарезервированного места для root на файловой системе:


```bash
sudo tune2fs -m 1 /dev/vda4
df -h /
```


Так я освободил немного дополнительного пространства, оставив только 1% под резерв.


![Скриншот 15 — выполнение tune2fs -m 1 и последующий df -h /](image15.png)

## Вывод


В ходе лабораторной работы я:


1. Освоил работу с **tmux**: научился делить окно на панели, открывать несколько окон, переключаться между ними и включать синхронизацию ввода для одновременного выполнения команд в нескольких панелях.


2. Научился пользоваться консольными утилитами мониторинга:
   - `free -h -s` — отслеживание использования оперативной памяти и swap;
   - `vmstat` — просмотр нагрузки на CPU, память, swap и IO;
   - `iostat` — анализ дисковой активности и задержек;
   - `watch df -h` — наблюдение за свободным местом на разделах в реальном времени.


3. На практике увидел, как `stress` влияет на систему: под нагрузкой сразу меняются показатели памяти и CPU, что помогает лучше понимать поведение живой системы.


4. Решил практическую задачу с переполненным диском:
   - нашёл крупный файл через `du`;
   - удалил файл и обнаружил, что место освободилось не полностью;
   - с помощью `lsof | grep deleted` нашёл процесс, держащий удалённый файл;
   - завершил этот процесс и полностью освободил место на корневом разделе;
   - при необходимости уменьшил резерв файловой системы с помощью `tune2fs`.


В результате я закрепил навыки мониторинга и диагностики, а также научился безопасно исправлять проблемы с переполнением диска без перезагрузки сервера, что важно для повседневной работы Linux‑администратора.
