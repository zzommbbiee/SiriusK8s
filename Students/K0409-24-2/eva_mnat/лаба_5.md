#Отчет о выполнении лабораторной работы по Kubernetes
По результатам предыдущего этапа было выявлено, что поды (Pods) не обладают достаточной автономностью для управления жизненным циклом приложения. Для обеспечения надежности и масштабирования было решено использовать ресурс Deployment.
##Блок 1: Работа с Deployment и ReplicaSet
Развертывание: Был применен готовый YAML-файл, описывающий систему из нескольких подов. Файл успешно применен к кластеру.
Мониторинг: В реальном времени отслеживался процесс создания подов. Статус развертывания (Deployment) проверен и соответствует ожидаемому.
ReplicaSet: Был проведен анализ связанного ресурса ReplicaSet. Данный компонент отвечает за поддержание заданного количества реплик подов (детали архитектуры были рассмотрены в рамках теории).
##Блок 2: Сервисы, балансировка и обновления
Доступ к приложению: Применен файл конфигурации сервиса. Получен внешний IP-адрес ноды (совпадает с IP виртуальной машины).
Балансировка нагрузки: Проведена проверка распределения трафика. Запросы успешно балансировались между тремя подами.
Rolling Update: Выполнено обновление образа контейнера.
Создан новый ReplicaSet и новые поды.
Старые поды удалены.
Результат: Трафик не прерывался (проверено через curl в отдельном терминале).
Откат изменений (Rollback): Проверена история ревизий. Выполнен откат к предыдущей версии. В истории развертывания номер версии уменьшился, что подтверждает успешность отката.
##Блок 3: Настройка Ingress
Инициализация: Включен Ingress Controller (компонент, реализующий правила маршрутизации).
Бэкенд-сервисы: Созданы два сервиса: простой текстовый и API-сервис.
Конфигурация Ingress: Применен YAML-файл Ingress.
Проверка доступности:
Настроен файл /etc/hosts для эмуляции доменного имени.
При прямой проверке через curl вывод отсутствовал.
Решение: Выполнено пробрасывание порта (port-forward) Ingress Controller на локальный порт 8080. Проверка через curl с явным указанием порта прошла успешно.
Статус подов Ingress: При проверке подов контроллера зафиксировано следующее состояние: 1 под в статусе Running, 2 пода в статусе Completed.
##Блок 4: Типы сервисов (Service Types)
Проведено сравнение способов открытия доступа к приложению:
ClusterIP: Создан сервис, доступный только внутри кластера.
Внутренний запрос: проходит успешно.
Внешний запрос: не проходит, так как для выделения внешнего IP требуется интеграция с облачным провайдером (в текущей среде отсутствует).
