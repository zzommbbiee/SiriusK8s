## Лабораторная работа №05 — Управление системными юнитами Systemd

Задача

Целью работы являлась диагностика и восстановление работоспособности системного сервиса script-server.service, находящегося в циклическом режиме перезапуска (Restart loop). Необходимо было выявить причину отказа, устранить её на уровне исполняемого сценария и провести аудит безопасности юнита.

Решение

Для решения задачи был применен комплексный подход, включающий анализ логов системы и аудит конфигурационных файлов.

Этапы выполнения:

Первичная диагностика: Использование systemctl status позволило определить текущее состояние юнита как failed.

Анализ журналов: С помощью journalctl -u script-server было установлено, что процесс завершается с кодом 1 (FAILURE) сразу после старта.

Прямое изучение исполняемого скрипта /opt/break_lab/script_server.sh выявило наличие директивы exit 1, провоцирующей принудительное завершение.

Восстановление: Внесение правок в исходный код скрипта (замена логики завершения на режим ожидания) и последующий перезапуск сервиса.

Проверка защищенности юнита встроенными средствами анализа systemd для оценки векторов потенциальных атак.

Команды, которые открыл для себя

systemd-analyze security — мощный инструмент для проведения автоматизированного аудита безопасности юнита. Позволяет увидеть «слабые места» в изоляции процесса.

systemd-cgtop — консольный монитор контрольных групп (Cgroups). Применяется для оперативного контроля потребления ресурсов (CPU/RAM) конкретными сервисами.

journalctl -b -u [unit] — фильтрация логов конкретного сервиса только за текущую сессию загрузки, что значительно ускоряет поиск ошибок.

systemctl reset-failed — очистка статуса «failed» для юнитов, что необходимо для приведения системы в исходное состояние после отладки.

Сложности
Наибольшую сложность представляло разграничение ответственности между конфигурацией самого юнита (.service файл) и логикой исполняемого скрипта. Изначально предполагалось, что проблема кроется в лимитах ресурсов (MemoryMax, CPUQuota), однако анализ логов показал, что сервис падает раньше, чем успевает достичь установленных ограничений. Также важным моментом стало понимание того, как RestartSec=2 влияет на плотность записей в системном журнале, что требовало умения быстро фильтровать повторяющиеся события.
