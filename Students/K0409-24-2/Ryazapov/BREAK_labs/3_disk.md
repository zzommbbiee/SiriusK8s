## Модуль 3 — Освободить диск
Задача

Необходимо было очистить переполненный корневой раздел (/) в условиях, когда виртуалку нельзя останавливать или перезагружать. Требовалось найти и удалить объемный файл, а также обнаружить и устранить «скрытую» утечку места, вызванную процессом, который продолжает удерживать удаленный файл в памяти. Итоговая цель — освободить минимум 6GB пространства.


Решение
Для выполнения задачи я делал по такому плану:

1. Первичная диагностика: С помощью df -h было подтверждено изменение раздела.
2. Поиск заполнителя: С помощью команды du -ahx / | sort -rh | head был локализован самый крупный файл в директории /var/lib/linux_break_lab/.
3. Удаление видимых данных: Файл был удален командой rm, однако мониторинг df показал, что место освободилось не полностью.
4. Поиск открытых дескрипторов: С помощью lsof | grep deleted был найден процесс break_lab_orphan_holder, который удерживал удаленный файл.
5. Освобождение блоков данных: После завершения процесса через kill -9 файловая система окончательно высвободила заблокированное пространство.
6. Оптимизация резерва: Для получения дополнительного свободного места был изменен процент резервирования блоков файловой системы с стандартных 5% до 1% с помощью утилиты tune2fs.

Команды которые открыл для себя
* du -ahx / — поиск файлов с ограничением только по одной файловой системе (флаг -x), что исключает мусор в выводе от /proc или /sys.
* lsof | grep deleted — поиск процессов, которые держат дескрипторы уже удаленных файлов, не давая системе очистить блоки данных.
* tune2fs -m 1 /dev/sdX — настройка процента зарезервированного места для суперпользователя. Позволяет мгновенно «вернуть» до 4-5% объема диска без удаления файлов.


Сложности
Команда rm не удаляет данные физически, пока файл открыт каким-либо процессом — это подтвердилось при повторной проверке df.
Идентификация блочного устройства: При использовании tune2fs возникла заминка с определением конкретного раздела. Команда df показывает точку монтирования /, но для утилиты обслуживания ФС потребовалось точно определить путь к устройству
