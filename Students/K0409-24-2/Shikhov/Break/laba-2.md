Модуль 2 — Починка сети
Задача
Нужно было восстановить работу сети в учебной системе. Скрипт «поломки» удалил стандартный маршрут (default route), подставил нерабочий DNS-сервер и запретил использовать внешние DNS-службы в конфигурации системы.
Итоговая цель — успешно выполнить запрос curl до внешнего ресурса (avito.ru).
Решение
Для исправления ситуации я выполнил следующие действия:
Запустил скрипт «поломки» network.sh, чтобы создать проблемную ситуацию для отработки навыков.
Отредактировал файл /etc/nsswitch.conf через nano. В строке hosts: добавил параметр dns, чтобы система могла запрашивать имена узлов через интернет, а не только искать их в локальном файле hosts.
Исправил настройки DNS в файле /etc/resolv.conf. Удалил неверный адрес 203.0.113.254 и прописал рабочий публичный DNS-сервер (например, 8.8.8.8).
Восстановил основной маршрут (маршрут по умолчанию) командой ip route add default, указав адрес шлюза моей сети.
Проверил доступность интернета командой curl -I https://www.avito.ru, получив успешный ответ от сервера.
Запись сессии: https://asciinema.org/a/DnLL8EqIT4TTJwnc
Команды которые открыл для себя
ip route add default via [IP] — команда для добавления маршрута по умолчанию. Без неё компьютер «не знает», куда отправлять пакеты, предназначенные для интернета
./etc/nsswitch.conf — важный файл, который определяет порядок поиска данных (паролей, имен хостов и т.д.). Теперь знаю, что если там нет слова dns, то интернет-адреса не будут резолвиться.
nano /etc/resolv.conf — редактирование списка DNS-серверов. Это первое место, куда стоит заглянуть, если «пинг по IP идет, а по именам — нет».
Сложности
Очередность правок: Сначала пытался добавить маршрут, но забыл про nsswitch.conf, поэтому curl всё равно выдавал ошибку. Важно помнить, что сеть — это комплекс настроек: и маршруты, и DNS-серверы, и системные правила поиска.
Права доступа: Все изменения в /etc/ требуют прав суперпользователя, поэтому приходилось постоянно использовать sudo.
