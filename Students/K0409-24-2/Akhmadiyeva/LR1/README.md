Команда ls -la /proc/$$/ns/ была использована для просмотра идентификаторов пространств имён текущего процесса оболочки, что позволило увидеть, какие типы пространств имён существуют и какие идентификаторы им присвоены. Команда lsns вывела список всех пространств имён, активных в системе, с указанием типов, идентификаторов и процессов, которые в них выполняются. Далее была выполнена команда sudo unshare --pid --fork --mount-proc /bin/bash, которая создала новое пространство имён процессов. Внутри этого окружения команда echo "Мой PID: $$" вывела идентификатор процесса, равный единице, что подтвердило изоляцию пространства имён, а команда ps aux отобразила только те процессы, которые принадлежат данному изолированному окружению. После выхода из этого окружения команда sudo unshare --net /bin/bash создала изолированное сетевое пространство, внутри которого команда ip link show показала наличие только интерфейса обратной петли, в то время как остальные сетевые интерфейсы хост-системы отсутствовали.
Команда ls /sys/fs/cgroup/ отобразила текущую иерархию контрольных групп, показав, что система использует вторую версию cgroup. Командой sudo mkdir /sys/fs/cgroup/mytest была создана новая контрольная группа, после чего с помощью команды echo "20000 100000" | sudo tee /sys/fs/cgroup/mytest/cpu.max для этой группы было установлено ограничение использования центрального процессора на уровне двадцати процентов. Для создания вычислительной нагрузки была запущена команда stress-ng --cpu 2 --timeout 30s &, которая выполнила стресс-тестирование центрального процессора в фоновом режиме. Идентификатор этого фонового процесса был добавлен в контрольную группу командой echo $! | sudo tee /sys/fs/cgroup/mytest/cgroup.procs. Затем команда cat /sys/fs/cgroup/mytest/cpu.stat вывела статистику использования процессора процессами внутри группы, включая общее время использования, количество периодов и количество превышений лимита. Команда top позволила визуально наблюдать, что загрузка процессора ограничена установленным значением. В завершение блока командой sudo kill $(cat /sys/fs/cgroup/mytest/cgroup.procs) была завершена нагрузка. В случае превышения лимита памяти механизм OOM-killer принудительно завершает процесс, потребляющий избыточный объём памяти, чтобы предотвратить дестабилизацию работы операционной системы.
Командами mkdir -p /tmp/myroot/{bin,lib,lib64,proc,dev} была создана структура каталогов для будущего корневого окружения. Команды cp /bin/bash /tmp/myroot/bin/ и cp /bin/ls /tmp/myroot/bin/ скопировали исполняемые файлы в новое окружение. Для определения зависимостей этих программ была использована команда ldd /bin/bash, которая показала необходимые библиотеки, включая libtinfo.so.6, libc.so.6 и ld-linux-x86-64.so.2. С помощью цикла for dep in $(ldd /bin/bash | awk '/=>/ {print $3}'); do cp --parents "$dep" /tmp/myroot/; done были скопированы все библиотеки с сохранением структуры каталогов, а команда cp /lib64/ld-linux-x86-64.so.2 /tmp/myroot/lib64/ добавила динамический загрузчик. После этого команда sudo chroot /tmp/myroot /bin/bash выполнила смену корневого каталога и запустила оболочку внутри изолированной файловой системы. Команда ls / внутри этого окружения вывела только те каталоги, которые были созданы вручную, а именно bin, lib, lib64 и proc, что подтвердило успешную изоляцию. Команда echo "Hello chroot!" продемонстрировала работоспособность окружения, после чего выход был выполнен командой exit.
Контрольные вопросы:
При выполнении команд unshare и chroot создаются изолированные окружения, которые существуют независимо от основной системы. Когда внутри такого окружения вводится команда exit, завершается только тот процесс, который был запущен в изолированном пространстве, но не затрагиваются процессы хост-системы. Это происходит потому, что механизмы изоляции, такие как пространства имён и смена корневого каталога, создают иерархическую структуру процессов, в которой дочерние процессы не могут влиять на родительские. Процесс, запущенный в изолированной среде, является потомком основного процесса оболочки, и его завершение не распространяется на предков. Кроме того, пространства имён обеспечивают виртуализацию системных ресурсов таким образом, что процессы внутри изолированной среды не имеют доступа к управлению процессами хост-системы и не могут их завершить. Таким образом, выход из изолированного окружения приводит лишь к освобождению ресурсов, занятых этим окружением, и возвращению управления родительской оболочке без каких-либо последствий для остальных процессов операционной системы.
В случае превышения установленного лимита памяти для контрольной группы или отдельного процесса вступает в действие механизм Out-Of-Memory Killer, который является компонентом ядра операционной системы. При достижении процессом или группой процессов установленного ограничения по памяти, ядро фиксирует факт превышения и принимает решение о необходимости освобождения ресурсов. OOM-killer анализирует текущее состояние системы, оценивая такие параметры, как объём потребляемой памяти, приоритет процессов, время их работы и важность для функционирования системы. На основе проведённого анализа выбирается один или несколько процессов для принудительного завершения, причём предпочтение обычно отдаётся процессам, потребляющим наибольшее количество памяти и имеющим наименьший приоритет. Выбранные процессы завершаются сигналом SIGKILL, который не может быть перехвачен или проигнорирован, после чего занимаемая ими память освобождается. В контексте контрольных групп, если лимит памяти превышен процессами, входящими в группу, OOM-killer может завершить один или несколько процессов внутри этой группы, либо, в зависимости от настроек, завершить все процессы группы, что позволяет предотвратить дестабилизацию работы всей операционной системы из-за нехватки памяти.
