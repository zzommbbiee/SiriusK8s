Первым этапом работы стала проверка состояния кластера и его компонентов. Команда kubectl get nodes -o wide вывела информацию об узле кластера. В выводе было видно, что единственная нода minikube находится в статусе Ready, имеет роль control-plane (что означает совмещение роли мастера и рабочей ноды), использует операционную систему Debian GNU/Linux 12 и среду выполнения контейнеров docker. Внутренний IP-адрес ноды составил 192.168.49.2, версия Kubernetes - v1.35.1. Статус Ready указывает на то, что kubelet на ноде успешно зарегистрировался в API-сервере и готов принимать поды. Детальный анализ ноды был выполнен с помощью kubectl describe node minikube | head -50. В выводе отобразились метки ноды, включая архитектуру процессора (amd64) и операционную систему (linux). Важным элементом вывода стали секции Conditions, показывающие текущее состояние ноды. Все условия - MemoryPressure, DiskPressure, PIDPressure - имели статус False, а условие Ready - True, что свидетельствует о нормальной работе узла. В разделе Capacity были указаны общие ресурсы ноды: 8 ядер CPU, около 15 ГБ оперативной памяти и 60 ГБ дискового пространства. Секция Allocatable показала ресурсы, доступные для размещения подов (с учётом резервирования для системных процессов).
Для просмотра системных компонентов была выполнена команда kubectl get pods -n kube-system. В пространстве имен kube-system были обнаружены следующие поды: coredns (обеспечивает DNS-резолвинг внутри кластера), etcd (хранилище данных кластера), kube-apiserver (точка входа для всех управляющих запросов), kube-controller-manager (управляет контроллерами), kube-scheduler (назначает поды на ноды), kube-proxy (обеспечивает сетевую связность), storage-provisioner (автоматически создает тома для хранилищ). Все поды находились в статусе Running, что подтверждает корректную работу control plane. Количество рестартов некоторых подов (например, storage-provisioner - 3 рестарта) связано с нормальным процессом инициализации кластера и не является признаком проблем.
Команда kubectl get componentstatuses (несмотря на предупреждение о deprecated) показала, что все три компонента control plane - controller-manager, scheduler и etcd - находятся в состоянии Healthy. Это подтверждает, что ключевые компоненты кластера функционируют без ошибок.
Доступ к файлам манифестов control plane был получен через вход в ноду minikube с помощью minikube ssh. Внутри ноды в директории /etc/kubernetes/manifests/ были обнаружены файлы kube-apiserver.yaml, kube-controller-manager.yaml, kube-scheduler.yaml. Эти YAML-файлы описывают статические поды, которые kubelet запускает на мастер-ноде для обеспечения работы control plane. Просмотр первых строк kube-apiserver.yaml показал параметры запуска API-сервера, включая адрес для публикации (192.168.49.2), режим авторизации (Node,RBAC), пути к сертификатам и параметры подключения к etcd.
С помощью kubectl api-resources | head -20 был выведен список доступных API-ресурсов кластера. В выводе представлены такие объекты, как pods, services, configmaps, namespaces и другие. Для каждого ресурса указаны краткие имена (например, po для pods, svc для services), версия API и признак принадлежности к пространству имен.
Версия клиента и сервера Kubernetes, полученная через kubectl version, составила Client Version v1.34.6 и Server Version v1.35.1. Разница в минорных версиях допустима, так как клиент может работать с сервером, версия которого отличается на ±1 минорную версию.
Создание первого пода было выполнено командой kubectl run nginx --image=nginx:alpine --port=80. Эта команда создала под с именем nginx, использующий образ nginx:alpine, и открыла порт 80 внутри контейнера. Наблюдение за созданием пода с помощью kubectl get pods -w показало последовательную смену статусов: сначала ContainerCreating, затем через несколько секунд Running. Это связано с тем, что kubelet на ноде скачивает образ, создает контейнер и запускает его.
После того как под перешел в статус Running, была выполнена команда kubectl exec -it nginx -- sh для входа внутрь контейнера. Внутри контейнера были выполнены следующие исследования:
1. hostname вывел nginx, что соответствует имени пода. В Kubernetes имя пода становится hostname контейнера.
2. cat /etc/hosts показал записи в файле hosts, включая локальную запись для IP-адреса пода (10.244.0.4). Этот IP-адрес был получен из сетевого плагина кластера и используется для взаимодействия между подами.
3. env | grep KUBE вывел переменные окружения, связанные с Kubernetes: KUBERNETES_SERVICE_HOST=10.96.0.1 и KUBERNETES_SERVICE_PORT=443. Это IP-адрес и порт API-сервера, которые автоматически внедряются в каждый под для обеспечения доступа к API кластера.
4. ps aux показал процессы внутри контейнера. Был виден процесс nginx master (PID 1) и несколько worker-процессов. Отсутствие других системных процессов (например, systemd) демонстрирует, что контейнер изолирован и запускает только указанные процессы.
5. ip addr вывел сетевые интерфейсы. Интерфейс eth0 имел IP-адрес 10.244.0.4/16, что соответствует сетевому пространству пода. Интерфейс lo - стандартная петля.
Логи пода, полученные командой kubectl logs nginx, содержали информацию о запуске nginx: выполнение скриптов из /docker-entrypoint.d/, настройка IPv6, запуск worker-процессов. Логи подтвердили, что nginx успешно запустился и готов принимать запросы. Команда kubectl describe pod nginx предоставила детальную информацию о поде: имя, пространство имен, ноду назначения, IP-адрес, состояние контейнера, события. В секции Events были зафиксированы ключевые этапы жизни пода: Scheduled (назначение на ноду minikube), Pulling (скачивание образа), Pulled (образ успешно скачан), Created (контейнер создан), Started (контейнер запущен). Время между Pulling и Pulled составило 15.8 секунды, что соответствует объему образа (около 62 МБ). Состояние Ready: True указывает на то, что под прошел readiness-проверку и готов принимать трафик.
Для более сложной конфигурации был создан YAML-манифест pod.yaml, описывающий под my-webserver с двумя контейнерами. Первый контейнер nginx является основным веб-сервером. Для него были заданы ресурсные ограничения: requests - 50m CPU и 32Mi памяти, limits - 100m CPU и 64Mi памяти. Такая конфигурация гарантирует, что контейнер получит минимум ресурсов и не сможет использовать больше установленного лимита. Также были настроены livenessProbe и readinessProbe. ReadinessProbe проверяет доступность веб-сервера через HTTP GET на корневой путь с задержкой 3 секунды и интервалом 5 секунд. LivenessProbe аналогична, но имеет большую начальную задержку (10 секунд) и интервал (10 секунд). Эти проверки позволяют контролировать готовность пода принимать трафик и автоматически перезапускать контейнер в случае неработоспособности. Второй контейнер log-sidecar на основе образа busybox выполняет бесконечный цикл, записывающий текущую дату и время в файл /var/log/access.log каждые 5 секунд. Этот контейнер использует общий том logs типа emptyDir, который монтируется в оба контейнера: в nginx по пути /var/log (не используется в данном примере) и в sidecar по тому же пути. emptyDir создает пустую директорию на ноде, которая существует пока жив под. Это демонстрирует механизм обмена данными между контейнерами внутри одного пода. Применение манифеста командой kubectl apply -f pod.yaml привело к созданию пода my-webserver. Проверка статуса kubectl get pods показала, что под находится в состоянии Running с готовностью 2/2 (оба контейнера запущены). Время создания составило около 19 секунд. Имена контейнеров в поде были получены с помощью kubectl get pod my-webserver -o jsonpath='{.spec.containers[*].name}', что вывело строку nginx log-sidecar. Это подтвердило, что под действительно содержит два контейнера. 
Логи sidecar-контейнера, извлеченные командой kubectl logs my-webserver -c log-sidecar, показали отсутствие вывода (так как контейнер пишет в файл, а не в stdout). При необходимости просмотра содержимого файла потребовался бы вход в контейнер. Вход в контейнер nginx был выполнен с помощью kubectl exec -it my-webserver -c nginx -- sh. Это позволило бы исследовать общий том, если бы была необходимость.
Полный YAML запущенного пода был получен командой kubectl get pod my-webserver -o yaml | head -60. В выводе видны добавленные Kubernetes поля: metadata.generation, metadata.resourceVersion, metadata.uid, status и другие. Интерес представляет секция spec.containers[0].livenessProbe, которая в исходном манифесте содержала только httpGet, initialDelaySeconds и periodSeconds. В итоговом YAML Kubernetes добавил значения по умолчанию: failureThreshold: 3, successThreshold: 1, timeoutSeconds: 1. Это демонстрирует, как control plane дополняет пользовательскую конфигурацию значениями по умолчанию.
Для проверки механизма самовосстановления был насильно завершен процесс nginx внутри контейнера командой kubectl exec my-webserver -c nginx -- kill 1. PID 1 - это основной процесс контейнера (nginx master). Его завершение означает остановку основного приложения. Наблюдение за статусом пода с помощью kubectl get pods -w показало, что после выполнения команды под остался в статусе Running, но количество рестартов (RESTARTS) увеличилось на 1. Это означает, что kubelet обнаружил завершение основного процесса и автоматически перезапустил контейнер в соответствии с политикой рестарта Always, установленной по умолчанию. При этом сам под как объект не был удален, так как его конфигурация остается в etcd, и kubelet на ноде поддерживает его в рабочем состоянии.
Финальная проверка kubectl get pod my-webserver подтвердила, что под находится в статусе Running с двумя готовыми контейнерами, а поле RESTARTS для пода увеличилось на 1 (стало 1). Это демонстрирует ключевой механизм Kubernetes: kubelet непрерывно следит за состоянием контейнеров и принудительно перезапускает их в случае сбоя, обеспечивая самовосстановление приложения без вмешательства оператора.
