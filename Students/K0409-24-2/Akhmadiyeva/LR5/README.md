В ходе выполнения работы был создан Deployment webapp с тремя репликами на основе образа nginxdemos/hello:plain-text. В файле deployment.yaml была настроена стратегия обновления RollingUpdate с параметрами maxSurge: 1 и maxUnavailable: 0, что гарантирует отсутствие простоев при смене версий. После применения манифеста командой kubectl apply все три пода успешно запустились, а созданный ReplicaSet подтвердил управление тремя экземплярами.
Затем был создан Service типа NodePort на порту 30080, обеспечивающий внешний доступ к подам. С помощью команды kubectl set image был выполнен rolling update до версии latest, а затем выполнен откат до предыдущей версии командой rollout undo. В процессе обновления и отката перебоев в работе не было, что подтверждается циклом curl-запросов, распределявших трафик между тремя подами. История деплойментов сохранила три ревизии.
Далее был включен Ingress Controller в minikube и создан Ingress-ресурс с правилами маршрутизации: путь / направляет трафик на webapp-svc, а путь /api - на сервис api-svc, который был развернут отдельно. После добавления записи в /etc/hosts команда curl webapp.local вернула ответ с именем сервера от одного из подов webapp, а curl webapp.local/api - строку «Hello from API». Это подтверждает корректную работу Ingress.
Для сравнения типов Service был создан ClusterIP-сервис webapp-clusterip. Проверка из временного пода Alpine показала, что этот сервис доступен только внутри кластера, в отличие от NodePort, который доступен снаружи по IP-адресу узла и статическому порту.
