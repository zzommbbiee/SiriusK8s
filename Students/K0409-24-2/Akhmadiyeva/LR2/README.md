Был создан «плохой» Dockerfile, который использовал базовый образ python:3.12, копировал в контейнер все файлы из текущей директории и устанавливал зависимости. Размер полученного образа составил более 1,6 ГБ. Базовый образ python:3.12 содержит полную операционную систему, компиляторы, заголовочные файлы и другие инструменты, необходимые для сборки пакетов, но не нужные для простого запуска приложения. Кроме того, в образ были скопированы все файлы из текущей директории, включая служебные и временные файлы, а кэш пакетного менеджера не был очищен после установки зависимостей. При запуске контейнера на основе этого образа приложение успешно работало на порту 5000, но итоговый образ оказался избыточно тяжелым.
Для оптимизации размера образа был разработан многокомпонентный (multistage) Dockerfile. На первой стадии сборки использовался образ python:3.12-slim, в который копировался только файл requirements.txt. Установка зависимостей выполнялась с флагом --user, что позволяло разместить пакеты в директории пользователя. На второй стадии применялся минимальный образ python:3.12-alpine, в который копировались только установленные ранее пакеты из первой стадии и файл приложения. Дополнительно были добавлены переменная окружения APP_VERSION со значением 2.0, создание непривилегированного пользователя appuser и инструкция для запуска приложения от его имени. Также был создан файл .dockerignore для исключения временных файлов, каталогов виртуального окружения и служебных директорий из контекста сборки. После такой оптимизации размер образа сократился до 81,4 МБ, что более чем в 20 раз меньше исходного варианта.
Запуск контейнера на основе оптимизированного образа выполнялся с ограничениями ресурсов: выделено 128 МБ оперативной памяти и 0,5 ядра процессора. Проброс портов был настроен таким образом, что приложение внутри контейнера продолжало слушать порт 5000, а на хосте было доступно на порту 5001. Контейнер был настроен на автоматический перезапуск в случае остановки с помощью опции --restart=unless-stopped. При проверке через docker stats подтвердилось, что ограничения ресурсов применяются корректно.
Для изучения структуры образов использовались команды docker history и docker inspect. В случае «плохого» образа было видно множество слоев большого размера, включающих системные библиотеки и инструменты. В «хорошем» образе количество слоев осталось таким же, но размер каждого из них значительно уменьшился, а в финальном образе отсутствуют слои, содержащие компиляторы и другие средства разработки. С помощью утилиты dive была выполнена визуальная оценка эффективности сборки: инструмент показал, какие слои занимают больше всего места и какие файлы были добавлены на каждом этапе.
На заключительном этапе работы подготовленный образ был опубликован в публичном репозитории Docker Hub. Выполнена авторизация через команду docker login, после чего образ был тегирован с указанием имени пользователя и названия репозитория flask-demo с тегом v1.0. Команда docker push успешно загрузила слои образа в облачный реестр. Для проверки корректности публикации локальная версия образа была удалена, затем выполнена его загрузка из Docker Hub и запуск нового контейнера с пробросом порта 5002 на хосте. Приложение успешно запустилось и оставалось доступным по указанному адресу.
