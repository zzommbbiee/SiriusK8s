На первом этапе работы было проведено исследование сетевых механизмов Docker. Была создана пользовательская bridge-сеть с именем app-network. В эту сеть был запущен контейнер с PostgreSQL, а затем запущен временный контейнер на базе Alpine Linux. Внутри временного контейнера была проверена доступность базы данных по имени контейнера с помощью команды ping db, а также проверен открытый порт 5432 с использованием утилиты nc. Результаты подтвердили, что контейнеры, подключенные к одной пользовательской сети, могут взаимодействовать друг с другом по именам, в отличие от контейнеров, работающих в изолированной среде без общей сети. Данное поведение объясняется тем, что каждая пользовательская сеть предоставляет собственный DNS-сервер, который разрешает имена контейнеров в их IP-адреса.
Для обеспечения сохранности данных был создан именованный том pgdata, который в дальнейшем использовался для хранения файлов базы данных PostgreSQL. Был запущен контейнер с PostgreSQL с монтированием созданного тома, после чего в базе данных была создана тестовая таблица и вставлена запись. Затем контейнер был удален, а на его основе запущен новый контейнер с тем же томом. Проверка показала, что созданные ранее данные сохранились, что демонстрирует возможность переноса данных между разными контейнерами и устойчивость информации к перезапускам. Команда docker volume inspect позволила определить физическое расположение тома на хост-системе.
Основная часть работы была посвящена созданию многоконтейнерного стека с использованием Docker Compose. Структура проекта была организована следующим образом: в папке backend размещены файлы Flask-приложения (app.py), файл зависимостей (requirements.txt) и Dockerfile для сборки образа. В папке frontend размещен конфигурационный файл nginx (nginx.conf), настраивающий проксирование запросов к backend-сервису и возвращающий простую HTML-страницу при обращении к корневому пути.
Flask-приложение реализует два маршрута. Маршрут /api/items подключается к базе данных PostgreSQL, выполняет запрос на выборку всех записей из таблицы items и возвращает результат в формате JSON. Маршрут /health возвращает статус «ok» и используется для проверки работоспособности сервиса. Приложение использует переменные окружения для настройки подключения к базе данных, что позволяет гибко изменять параметры соединения без модификации кода.
Файл docker-compose.yml описывает три сервиса. Сервис db использует официальный образ PostgreSQL версии 16 на базе Alpine Linux, задает переменные окружения для создания базы данных и пользователя, монтирует том pgdata для сохранения данных, а также содержит healthcheck, проверяющий готовность базы данных к приему подключений с помощью команды pg_isready. Сервис backend собирается из Dockerfile, расположенного в папке ./backend, получает переменные окружения с параметрами подключения к базе данных, зависит от сервиса db с условием service_healthy, что гарантирует запуск backend только после полной готовности базы данных. Healthcheck для backend использует команду wget для проверки эндпоинта /health. Сервис frontend использует официальный образ nginx на базе Alpine, пробрасывает порт 8081 хост-системы на 80 порт контейнера, монтирует конфигурационный файл nginx и зависит от сервиса backend с условием service_healthy.
В ходе сборки и запуска стека было выполнено создание образов, развертывание контейнеров и настройка сети. Команда docker compose up -d --build успешно собрала образ backend, загрузила необходимые образы для PostgreSQL и nginx, создала сеть и том, а также запустила все три контейнера. Проверка статуса с помощью docker compose ps показала, что все контейнеры находятся в состоянии healthy, что подтверждает успешное прохождение healthcheck для каждого сервиса.
Для проверки работоспособности всего стека в базу данных были добавлены тестовые записи с использованием команды docker compose exec db psql. Выполнение запроса curl localhost:8081/api/items (порт был изменен на 8081 для избежания конфликта) вернуло корректный JSON-ответ, содержащий три записи: «apple», «banana» и «cherry». Таким образом, была подтверждена полная цепочка взаимодействия: клиентский запрос поступает на nginx, который проксирует его на backend-сервис, backend подключается к базе данных и возвращает результат через nginx обратно клиенту.
На заключительном этапе была выполнена операция масштабирования backend-сервиса с помощью команды docker compose up -d --scale backend=3. После выполнения данной команды количество экземпляров backend увеличилось до трех, что было подтверждено выводом docker compose ps. При этом все три экземпляра успешно прошли healthcheck и работали параллельно, обрабатывая входящие запросы. Масштабирование позволяет распределять нагрузку между несколькими экземплярами приложения, повышая отказоустойчивость и производительность системы.
По завершении работы стек был остановлен и удален с использованием команд docker compose down и docker compose down -v, причем второй вариант также удалил созданный том с данными. Была выполнена очистка неиспользуемых ресурсов с помощью docker system prune, что позволило освободить дисковое пространство.
