В ходе выполнения лабораторной работы был создан namespace rbac-demo и настроен ServiceAccount app-reader с ролью pod-reader, ограничивающей права только на просмотр подов. Проверка прав с помощью команды kubectl auth can-i показала, что учетная запись может получать список подов в своем namespace, но не имеет права их удалять, а также не может просматривать поды в других namespace, что подтверждает корректную реализацию принципа минимальных привилегий. Под rbac-test был запущен контейнер с назначенным ServiceAccount; внутри пода операции чтения подов выполнялись успешно, попытка удаления завершалась ошибкой Forbidden, что полностью соответствует заданной политике RBAC.
Для проверки сетевой изоляции в namespace netpol-demo были развернуты поды frontend, backend и database с соответствующими метками. Были применены NetworkPolicy, ограничивающие входящий трафик к backend только от frontend, а к database - только от backend. Несмотря на настройку правил и перезапуск подов, тесты показали, что frontend все равно может подключаться к database, что свидетельствует о некорректной фильтрации трафика либо особенностях используемого CNI-плагина. TLS-сертификаты для frontend были успешно сгенерированы и подключены через Ingress, соединение проверено с использованием CA. Четвертый пункт работы, запуск и проверка Falco, выполнен не был, так как Falco в кластере отсутствует.
