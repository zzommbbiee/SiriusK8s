# Отчет по лабораторной работе: Docker Networking, Volumes и Docker Compose

## 1. Чему научился

В ходе выполнения лабораторной работы освоены следующие практические навыки работы с Docker:

**Работа с сетями (Docker Networking)**
- Создание изолированных пользовательских сетей с помощью `docker network create --driver bridge`
- Понимание принципов работы встроенного DNS Docker, позволяющего контейнерам в одной сети обращаться друг к другу по имени сервиса
- Диагностика сетевых подключений между контейнерами с использованием утилит `ping` и `nc`
- Проверка конфигурации сети через `docker network inspect`

**Управление данными (Docker Volumes)**
- Создание и использование именованных томов командой `docker volume create`
- Монтирование томов в контейнеры через флаг `-v`
- Демонстрация персистентности данных: удаление контейнера не влияет на данные в томе, которые восстанавливаются при запуске нового контейнера
- Локализация физического расположения тома через `docker volume inspect`

**Оркестрация контейнеров (Docker Compose)**
- Написание файла `docker-compose.yml` для описания многосервисного приложения (frontend, backend, database)
- Настройка зависимостей между сервисами с использованием `depends_on` и `healthcheck`
- Подъем и остановка всего стека командой `docker compose up -d` и `docker compose down`
- Масштабирование сервисов (`--scale backend=3`) и понимание ограничений (конфликт с жестко заданным `container_name`)
- Просмотр состояния сервисов (`docker compose ps`) и логов (`docker compose logs -f`)
- Выполнение команд внутри контейнеров стека через `docker compose exec`

## 2. Проблемы и способы их решения

| Проблема | Решение |
|----------|---------|
| **Ошибка доступа к `/var/run/docker.sock`** | Использована команда `sudo chmod` для корректировки прав на директории. Выполнена диагностика через `namei -l` для отслеживания полного пути и прав доступа к символьным ссылкам. |
| **Отсутствие команды `docker-compose`** | В системе не был установлен Docker Compose. Предложена установка пакетов `docker-compose-2.39.1` и `docker-compose-switch`. После установки команда `docker compose` (без дефиса) стала доступна. |
| **Ошибка синтаксиса: `docker compose up -d --build`** | Пользователь указал оба флага в неверном порядке. Правильный синтаксис: `docker compose up -d --build`. |
| **Предупреждение при масштабировании backend** | В файле `docker-compose.yml` для сервиса `backend` был жестко прописан `container_name: compose-lab-backend-1`. При попытке масштабирования до 3 экземпляров Docker не может создать контейнеры с одинаковыми именами. **Решение:** удаление или комментирование строки `container_name` из файла конфигурации. |
| **Неоднозначная проверка healthcheck в Alpine** | Для образа `nginx:alpine` команда `wget` для проверки healthcheck может работать некорректно. Предложена альтернатива с использованием `python3 -c "import urllib.request; urllib.request.urlopen('http://localhost:5000/health')"`. |
| **Read-only файловая система при конфигурации nginx** | В логах появилось предупреждение о невозможности модифицировать `/etc/nginx/conf.d/default.conf`. Проблема решена монтированием собственного конфигурационного файла `nginx.conf` в режиме `:ro` (read-only). |

## 3. Ответы на контрольные вопросы

**1. Как Docker разрешает имена контейнеров в сети?**

Docker использует встроенный DNS-сервер. Когда создается пользовательская сеть, контейнеры в ней автоматически получают возможность обращаться друг к другу по имени контейнера или по имени сервиса (в Docker Compose). DNS-резолвинг работает только внутри одной сети. При выполнении команды `ping db` из контейнера alpine в сети `app-network` Docker возвращает IP-адрес контейнера с именем `db` (в примере — 172.19.0.2).

**2. В чем разница между `bridge`, `host` и `none` драйверами сетей?**

| Драйвер | Описание |
|---------|----------|
| `bridge` | Сеть по умолчанию. Контейнеры изолированы, получают собственные IP-адреса, связь с хостом происходит через NAT. Подходит для большинства сценариев. |
| `host` | Контейнер использует сетевой стек хоста напрямую (без изоляции). IP-адрес контейнера совпадает с IP хоста. Увеличивает производительность, но снижает безопасность. |
| `none` | Контейнер не имеет сетевого интерфейса, кроме loopback. Полная сетевая изоляция. |

**3. Где физически хранятся данные Docker volumes на хосте?**

По умолчанию — в директории `/var/lib/docker/volumes/`. В ходе лабораторной работы выполнена команда `docker volume inspect pgdata`, которая показала точный путь: `/home/var_new/lib/docker/volumes/pgdata/_data`. Каждый том представлен отдельной директорией, внутри которой находятся данные контейнера в неизменном виде.

**4. Какая основная проблема возникает при масштабировании сервиса с `container_name`?**

Проблема: `container_name` задает жесткое, уникальное в пределах хоста имя контейнера. При масштабировании (например, `--scale backend=3`) Docker пытается создать несколько контейнеров с одинаковым именем, что невозможно. Решение: не указывать `container_name` в сервисе, который планируется масштабировать. Docker Compose сам генерирует имена с добавлением числового суффикса (`backend-1`, `backend-2`, `backend-3`).

**5. Для чего используются `healthcheck` и `depends_on` в Docker Compose?**

- **`depends_on`** — определяет порядок запуска сервисов. Сервис, указанный в `depends_on`, запускается раньше зависимого от него сервиса. Однако сам по себе `depends_on` не проверяет готовность сервиса к работе (например, не ждет, пока PostgreSQL примет соединения).

- **`healthcheck`** — определяет команду для проверки работоспособности сервиса. В сочетании с `depends_on: condition: service_healthy` позволяет запускать следующий сервис только после того, как зависимый сервис прошел проверку здоровья. В работе использованы: для БД — `pg_isready`, для backend — `wget http://localhost:5000/health`.

**6. Как проверить работоспособность всего стека приложения?**

Команды для проверки:

```bash
# Статус всех сервисов
docker compose ps

# Логи всех сервисов в реальном времени
docker compose logs -f

# Проверка конечной точки через curl
curl localhost:8080/api/items

# Выполнение SQL-запросов к БД
docker compose exec db psql -U user -d mydb -c "SELECT * FROM items;"
