# Отчет по лабораторной работе: Kubernetes: ConfigMap, Secret, PersistentVolume

## 1. Чему научился

В ходе выполнения лабораторной работы изучены и отработаны следующие возможности Kubernetes:

- Создание ConfigMap из литералов (`kubectl create configmap --from-literal`) и из файла (`--from-file`).
- Передача ConfigMap в под тремя способами:
  - все ключи как переменные окружения через `envFrom`;
  - конкретный ключ с переименованием через `env` + `configMapKeyRef`;
  - монтирование ConfigMap как файла в volume.
- Создание Secret из литералов и просмотр его содержимого в формате base64 (с демонстрацией, что это не шифрование).
- Понимание уровня безопасности Secret по умолчанию – только base64‑кодирование, настоящее шифрование в etcd отсутствует.
- Работа с PersistentVolumeClaim (PVC): создание PVC, привязка к поду PostgreSQL, проверка сохранения данных после удаления пода (благодаря PVC и автоматическому пересозданию пода Deployment).
- Использование `envFrom` для Secret – удобная загрузка всех секретов в окружение контейнера.
- Практическое подтверждение – после удаления пода PostgreSQL данные внутри PVC остаются нетронутыми, новый под успешно их читает.

## 2. Проблемы и как решены

| Проблема | Решение |
|----------|---------|
| При создании PVC статус `Pending` | Проверен `storageClassName`: для minikube указан `standard`, для k3s – `local-path`. В манифесте задан правильный класс. |
| В логах пода `config-demo` не отображается содержимое файла `/etc/config/nginx.conf` | Проверено, что ConfigMap `nginx-conf` создан из файла `nginx.conf`, а в манифесте пода правильно указаны `volumeMounts` и `volumes` с именем `configMap`. После пересоздания пода файл смонтирован корректно. |
| После удаления пода PostgreSQL данные не отображаются | Нужно дождаться, пока Deployment пересоздаст под (`kubectl get pods -w`). Затем выполнить `exec` с правильным именем нового пода. Данные оказались на месте, так как PVC не удалялся. |
| Команда `psql` внутри контейнера PostgreSQL выдает ошибку подключения | Исправлено указанием имени БД и пользователя: `psql -U pguser -d mydb -c "SELECT ..."`. Переменные окружения из Secret `postgres-secret` переданы через `envFrom`. |
| При просмотре Secret через `kubectl get secret -o yaml` данные видны в base64, но в etcd хранятся открыто | Без `EncryptionConfiguration` etcd хранит значения в открытом виде. Для демонстрации требуется доступ к etcd, что в minikube ограничено, но концепция усвоена. |


## 3. Ответы на вопросы

**Вопрос:** *Почему Secret в Kubernetes по умолчанию небезопасен и что делать?*

**Ответ:**  
По умолчанию Secret в Kubernetes **не шифруется** – данные только кодируются в base64. Это не является шифрованием, любой, кто имеет доступ к API или дампу etcd, может декодировать значение командой `base64 -d`. Более того, etcd (хранилище всех объектов кластера) по умолчанию хранит Secret'ы в открытом виде на диске.

**Что делать для повышения безопасности:**

1. **Включить шифрование в etcd** – настроить `EncryptionConfiguration` с провайдерами `aescbc`, `aesgcm` или `kms`. Это заставит kube-apiserver шифровать Secret'ы перед записью в etcd.
2. **Использовать внешние системы управления секретами** – HashiCorp Vault, AWS Secrets Manager, GCP Secret Manager, Azure Key Vault.
3. **Применять контроллеры типа Sealed Secrets** – зашифрованный Secret можно безопасно хранить в Git.
4. **Ограничивать доступ к Secret'ам через RBAC** – разрешать чтение только необходимым сервисным аккаунтам и пользователям.
5. **Использовать шифрование на стороне приложения** – хранить в Secret'е только зашифрованные блобы, а ключи расшифровки держать вне кластера.

> **Вывод:** базовый механизм Secret подходит для хранения некритичных данных (например, паролей dev-среды), но для production требует обязательного дополнительного шифрования и/или интеграции с внешним vault-решением.
