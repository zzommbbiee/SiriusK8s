# Отчёт по лабораторной работе: Безопасность Kubernetes (RBAC, NetworkPolicy, Falco)

## 1. Чему удалось научиться

В ходе выполнения лабораторной работы осваиваются ключевые механизмы безопасности Kubernetes на практике.

**RBAC (Role-Based Access Control)**  
Выполняется создание ServiceAccount, Role и RoleBinding. Отрабатывается ограничение прав сервисного аккаунта только на чтение подов и логов в пределах одного namespace. Проверка прав выполняется командой `kubectl auth can-i` — это удобный способ быстро определить наличие разрешения у пользователя или SA. Запускается под с этим ServiceAccount, после чего внутри контейнера проверяется, что `kubectl` может только просматривать поды, но не удалять их, и не видит поды в другом namespace.

**NetworkPolicy**  
Осваивается работа сетевой изоляции на уровне L3/L4. Создаётся политика `default-deny-ingress`, запрещающая весь входящий трафик ко всем подам. Затем добавляются разрешающие политики: `allow-frontend-ingress` (фронтенд может принимать запросы из любого источника), `allow-backend-from-frontend` (бэкенд — только от фронтенда) и `allow-database-from-backend` (база данных — только от бэкенда). С помощью `wget` проверяется, что фронтенд больше не может напрямую обращаться к базе, а бэкенд — может. Наглядно демонстрируется применение политик и ограничение трафика между подами.

**TLS/SSL с OpenSSL и Ingress**  
Осваивается создание собственного центра сертификации (CA): генерируется приватный ключ и корневой сертификат. Затем создаются ключ и CSR для веб-приложения, подписанные своим CA с указанием Subject Alternative Names (SAN). Проверяется цепочка доверия. Полученный сертификат и ключ импортируются в Kubernetes как Secret типа `kubernetes.io/tls`. Настраивается Ingress с этим Secret, включается SSL-редирект. Добавляется запись в `/etc/hosts`, после чего проверяется, что `curl` с указанием созданного CA успешно открывает сайт по HTTPS. Также отрабатывается получение сертификата из Secret и просмотр его полей через `kubectl` и `openssl`.

**Falco**  
Выполняется установка Falco через Helm (предварительно устанавливается сам Helm). Запускается мониторинг, и наблюдается, как Falco детектирует запуск оболочки внутри контейнера и чтение чувствительных файлов (например, `/etc/shadow`). Понимается принцип работы runtime security для обнаружения аномальной активности.

## 2. Возникшие проблемы и способы их решения

**1. `kubectl exec` в под rbac-test выдает `container not found`**  
Сразу после создания пода попытка войти в него приводит к ошибке «container not found». Причина — контейнер ещё не запущен (статус не `Running`). Выполняется ожидание в несколько секунд, проверяется `kubectl get pod rbac-test -n rbac-demo`. После перехода пода в статус `Running` команда `exec` отрабатывает нормально. В дальнейшем учитывается время, необходимое для инициализации контейнера.

**2. Minikube не включает Calico как addon**  
При попытке `minikube addons enable calico` возникает сообщение «calico is not a valid addon». Выясняется, что в используемой версии minikube Calico больше не входит в стандартный набор аддонов. Проблема решается пересозданием кластера с явным указанием CNI:  
`minikube delete` → `minikube start --cni=calico --driver=docker`.  
После этого сетевые политики начинают работать корректно. Однако после пересоздания кластера требуется повторно развернуть все поды и политики — этот момент учитывается при работе с чистым кластером.

**3. Поды в состоянии `ContainerCreating` после перезапуска кластера**  
После старта нового minikube с Calico поды некоторое время находятся в состоянии `ContainerCreating`. Выполняется ожидание около минуты, пока CNI настраивается, после чего поды переходят в `Running`. При необходимости можно было бы удалить и пересоздать поды или проверить логи Calico.

**4. `curl` не подключается к `webapp.local`**  
После создания Ingress и добавления записи в `/etc/hosts` команда `curl` выдаёт ошибку подключения. Выясняется, что не включён Ingress Controller в minikube. Выполняется команда:  
`minikube addons enable ingress`  
После ожидания запуска подов ingress-nginx подключение устанавливается. Важно помнить, что Ingress Controller требуется отдельно — сам по себе ресурс Ingress не маршрутизирует трафик.

**5. Команда `helm` не найдена при установке Falco**  
В системе отсутствует Helm. Решается через пакетный менеджер: принимается предложение терминала установить пакет `helm` (в дистрибутиве на основе Fedora/RHEL). После установки Helm работает, и Falco устанавливается успешно. При отсутствии автоматического предложения можно было бы скачать бинарник с официального сайта или использовать `curl` для установки.

**6. Ошибка в jsonpath при просмотре сертификата из Secret**  
В одном из примеров команды встречается опечатка: `'{.data.tls}.crt}'` вместо `'{.data.tls\.crt}'` (точка перед `crt` должна быть экранирована). Из-за этого команда не выполняется. Исправляется на правильный jsonpath, после чего секрет успешно декодируется.

## 3. Результаты проверок (ответы на вопросы)

Ниже приведены результаты команд, которые требовалось показать. Все проверки выполняются успешно.

| Проверка | Ожидаемый результат | Фактический результат |
|----------|---------------------|------------------------|
| `kubectl auth can-i list pods -n rbac-demo --as=system:serviceaccount:rbac-demo:app-reader` | yes | **yes** |
| `kubectl auth can-i delete pods -n rbac-demo --as=system:serviceaccount:rbac-demo:app-reader` | no | **no** |
| `kubectl exec frontend -n netpol-demo -- wget -qO- --timeout=3 database-svc` | timeout (отказ соединения) | **таймаут** (NetworkPolicy блокирует) |
| `kubectl exec backend -n netpol-demo -- wget -qO- --timeout=3 database-svc` | 200 OK, HTML страница | **200 OK**, выводится страница nginx |
| `openssl verify -CAfile ca.crt webapp.crt` | `webapp.crt: OK` | **OK** |
| `curl --cacert ca.crt https://webapp.local` | HTML-ответ от nginx (фронтенд) | **получена страница nginx**, TLS установлен |

**Дополнительные проверки:**

- Внутри пода `rbac-test` команда `kubectl get pods -n rbac-demo` работает, а `kubectl delete pod rbac-test -n rbac-demo` выдаёт `Error from server (Forbidden)`.  
- `kubectl get networkpolicies -n netpol-demo` показывает четыре политики: `default-deny-ingress`, `allow-frontend-ingress`, `allow-backend-from-frontend`, `allow-database-from-backend`.  
- `openssl s_client -connect webapp.local:443 -CAfile ca.crt` возвращает `Verify return code: 0 (ok)` и показывает subject `CN=webapp.local`, issuer — `CN=SiriusLab Root CA`.  
- Срок действия сертификата из секрета составляет 365 дней (проверяется через `openssl x509 -enddate`).  

**Falco:**  
После запуска Falco и выполнения `kubectl exec -it frontend -n netpol-demo -- sh` в логах Falco появляется предупреждение о запуске оболочки в контейнере. При попытке `cat /etc/shadow` генерируется ещё один алерт о чтении чувствительного файла. Таким образом, Falco успешно обнаруживает подозрительные действия.
