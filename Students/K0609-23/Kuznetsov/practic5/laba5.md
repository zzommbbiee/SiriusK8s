# Отчёт по лабораторной работе: Kubernetes (Deployment, Service, Ingress)

## 1. Чему научился

В ходе выполнения данной лабораторной работы осваиваются следующие навыки:

- **Создание Deployment с несколькими репликами**  
  Описывается YAML-манифест, где указывается `replicas: 3`, стратегия обновления `RollingUpdate` с параметрами `maxSurge: 1` и `maxUnavailable: 0`. Это гарантирует, что во время обновления не происходит отказа в обслуживании.

- **Выполнение rolling update без даунтайма**  
  С помощью команды `kubectl set image deployment/webapp webapp=nginxdemos/hello:latest` обновляется образ контейнера. Параллельно запущенный цикл `curl` показывает, что ответы от разных подов меняются, но ошибки отсутствуют – обновление происходит незаметно для пользователя.

- **Откат на предыдущую версию**  
  Команда `kubectl rollout undo deployment/webapp` возвращает деплоймент к предыдущей ревизии. Команда `kubectl rollout history deployment/webapp` позволяет просмотреть историю изменений и убедиться, что откат отработал корректно.

- **Настройка Service типов ClusterIP и NodePort**  
  Создаётся Service с `type: NodePort` и указанием `nodePort: 30080`. Через `selector` сервис связывается с подами. Для проверки доступа снаружи кластера используется IP-адрес ноды (например, `minikube ip`) и порт 30080.

- **Использование Ingress для маршрутизации трафика**  
  Включается Ingress Controller (в minikube – `minikube addons enable ingress`). Создаётся Ingress-правило, направляющее запросы на `webapp.local/` к основному сервису `webapp-svc`, а запросы на `/api` – к сервису `api-svc`. Добавляется запись в `/etc/hosts` для разрешения доменного имени.

- **Различие типов Service на практике**  
  Экспериментально проверяется, что ClusterIP доступен только внутри кластера (например, из пода `alpine` через `wget`), NodePort позволяет обращаться снаружи по IP любой ноды и указанному порту, а LoadBalancer (в облачных средах) выдаёт внешний IP.

## 2. Проблемы и их решения

В процессе работы возникают следующие трудности и находятся способы их устранения:

- **Ingress не отвечает на `curl webapp.local`**  
  *Проблема:* после создания Ingress и включения аддона запрос не доходит до приложения.  
  *Решение:* выясняется, что доменное имя `webapp.local` не резолвится. Выполняется команда `echo "$(minikube ip) webapp.local" | sudo tee -a /etc/hosts`. После добавления записи в файл hosts curl начинает работать.

- **При rolling update появляются ошибки 502 (Bad Gateway)**  
  *Проблема:* в манифесте Deployment отсутствует `readinessProbe`. Новый под начинает принимать трафик до того, как полностью запустится, а старый под уже завершается.  
  *Решение:* в спецификацию контейнера добавляется `readinessProbe` с проверкой пути `/` на порту 80. Kubernetes начинает ждать готовности пода перед перенаправлением трафика, даунтайм исчезает.

- **Невозможно получить IP ноды для проверки NodePort**  
  *Проблема:* команда `kubectl get nodes -o wide` показывает внутренние адреса, недоступные с хоста в minikube.  
  *Решение:* используется команда `minikube ip`, которая возвращает корректный IP (например, `192.168.49.2`). Через этот IP и порт 30080 выполняется проверка.

- **При запросе `curl webapp.local/api` возвращается 404**  
  *Проблема:* Ingress передаёт полный путь `/api` в сервис `api-svc`, а контейнер `hashicorp/http-echo` ожидает только корневой путь.  
  *Решение:* в манифест Ingress добавляется аннотация `nginx.ingress.kubernetes.io/rewrite-target: /`. Теперь запрос `/api` переписывается в `/` при проксировании на бэкенд.

## 3. Ответы на вопросы

> **Объяснить разницу между ClusterIP / NodePort / LoadBalancer**

- **ClusterIP** – стандартный тип Service. Ему присваивается внутренний IP-адрес, доступный **только внутри кластера Kubernetes**. Снаружи (с хостовой машины или из интернета) обратиться к такому сервису невозможно. Используется для взаимодействия между компонентами внутри кластера.

- **NodePort** – расширяет функциональность ClusterIP. На каждой ноде кластера открывается определённый порт (обычно в диапазоне 30000–32767). Любой запрос, приходящий на `IP_ноды:NodePort`, перенаправляется на соответствующий Service. Удобен для тестирования или в средах без облачного балансировщика.

- **LoadBalancer** – работает поверх NodePort, но дополнительно интегрируется с внешним облачным провайдером (AWS, GCP, Yandex Cloud и т.д.). Провайдер автоматически создаёт балансировщик нагрузки с публичным IP-адресом. Весь трафик с этого IP направляется на Service. В локальных средах (minikube, kind) этот тип без дополнительных инструментов (например, MetalLB) остаётся в статусе Pending.

> **В чём конкретно разница ClusterIP и NodePort?**

Главное отличие – **доступность снаружи кластера**:

| Характеристика | ClusterIP | NodePort |
|----------------|-----------|----------|
| Доступ изнутри кластера | Да | Да |
| Доступ с хостовой машины | Нет (без port-forward) | Да |
| Открытый порт на каждой ноде | Нет | Да (30000–32767) |
| Типичное применение | Внутреннее взаимодействие микросервисов | Тестирование, отладка, простые сценарии |

Проверка на практике:
- Из пода (`kubectl run test --rm -it --image=alpine -- sh`) выполняется `wget -qO- webapp-clusterip` – ответ получен.
- С хостовой машины `curl 192.168.49.2:30080` – ответ получен (NodePort).
- С хостовой машины `curl <ClusterIP-адрес>` – соединение не устанавливается, так как этот IP не маршрутизируется за пределами кластера.

> **Почему при rolling update не было даунтайма?**

Даунтайм отсутствует благодаря:
- Стратегии `RollingUpdate` с `maxUnavailable: 0` – старые поды не удаляются, пока не будут готовы новые.
- `maxSurge: 1` – разрешён запуск одного дополнительного пода сверх реплик.
- Наличию `readinessProbe` – новый под начинает получать трафик только после успешной проверки готовности.
- Встроенному механизму балансировки Service – запросы распределяются между старыми и новыми подами в процессе обновления.
