# Docker: образы, Dockerfile, запуск

## 1. Чему научился

- **Сборка образов:** научился писать `Dockerfile` (инструкции `FROM`, `WORKDIR`, `COPY`, `RUN`, `CMD`), использовать `.dockerignore` для исключения ненужных файлов.
- **Многоэтапная сборка:** освоил технику `multistage build`, позволяющую уменьшить размер финального образа за счёт отделения этапа установки зависимостей.
- **Управление ресурсами:** применил ограничения по памяти (`--memory="128m"`) и CPU (`--cpus="0.5"`) при запуске контейнера, проверил их через `docker stats`.
- **Анализ образов:** изучил `docker history` для просмотра слоёв, `docker inspect` для детальной информации, `docker export` для извлечения файловой системы контейнера.
- **Работа с реестром:** выполнил авторизацию в Docker Hub, тегирование образа (`docker tag`) и публикацию (`docker push`), а также загрузку образа (`docker pull`).

## 2. Проблемы и их решение

- **Ошибка доступа к Docker API:** при попытке выполнить `docker build` получил сообщение  
  `permission denied while trying to connect to the docker API at unix:///var/run/docker.sock`.  
  *Решение:* использовал `sudo` для всех команд Docker  `docker`.

- **Невозможность установить `dive` через `dpkg`:** система не поддерживала `dpkg` (используется `dnf`).  
  *Решение:* скачал архив `.tar.gz`, распаковал бинарный файл и переместил его в `/usr/local/bin/`, после чего выполнил `hash -r` для обновления кэша команд.

## 3. Ответы на контрольные вопросы

**1. Почему образ `myapp:bad` получился таким большим (~1,12 ГБ)?**  
- Использован базовый образ `python:3.12`, который основан на Debian и содержит множество системных библиотек и утилит, не нужных для запуска простого Flask-приложения.  
- Отсутствие `.dockerignore` привело к копированию всех файлов из контекста сборки (включая временные файлы, историю Git и т.д.).  
- Зависимости устанавливались непосредственно в образ, что также увеличило его размер.  

**2. Как многоэтапная сборка уменьшила размер образа?**  
В многоэтапной сборке зависимости устанавливаются в промежуточном образе (builder), а в финальный образ копируются только необходимые файлы (`/root/.local`). Финальный образ использует минимальный базовый образ `python:3.12-alpine`, что привело к сокращению размера с 1,12 ГБ до 53,6 МБ.

**3. Как проверить, что ограничения ресурсов работают?**  
После запуска контейнера с флагами `--memory` и `--cpus` команда `docker stats` показывает текущее потребление и установленные лимиты. Например, в выводе `MEM USAGE / LIMIT` отображается `22.33MiB / 128MiB`, что подтверждает действие ограничения памяти.

## Результаты выполнения команд

| Команда | Результат |
|---------|----------|
| `sudo docker images myapp` | `myapp:bad    1.12GB`<br>`myapp:good   53.6MB` |
| `sudo docker history myapp:good` | Список слоёв с размерами (начиная с `COPY app.py` и `COPY /root/.local`) |
| `sudo docker stats app-good --no-stream` | `MEM USAGE / LIMIT: 22.33MiB / 128MiB`, `CPU %: 0.02%` |
| `sudo docker push ratedo/flask-demo:v1.0` | Успешная отправка всех слоёв в Docker Hub |
| `sudo docker pull ratedo/flask-demo:v1.0` | Успешная загрузка образа из реестра |

**URL образа на Docker Hub:** `https://hub.docker.com/repository/docker/ratedo/flask-demo/general`
