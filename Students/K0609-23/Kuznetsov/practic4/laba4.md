## 1. Чему научилcя

### Блок 1 — Состояние кластера
* **Анализ узлов:** Выполняется проверка статуса узлов через `kubectl get nodes -o wide`. Подтверждается готовность ноды `minikube` (статус `Ready`, версия `v1.35.1`).
* **Диагностика Control Plane:** Осуществляется проверка состояния системных компонентов через `kubectl get componentstatuses`. Все основные службы (`scheduler`, `controller-manager`, `etcd-0`) находятся в состоянии `Healthy`.
* **Исследование конфигурации:** Внутри ВМ minikube проводится осмотр директории `/etc/kubernetes/manifests/`. Изучаются манифесты статических подов, которые запускаются напрямую `kubelet`'ом.

### Блок 2 — Работа с первым Pod
* **Императивный запуск:** Отрабатывается запуск пода `nginx` из образа `nginx:alpine`.
* **Сетевая изоляция:** С помощью `kubectl exec` и команды `ip addr` внутри контейнера подтверждается наличие выделенного IP-адреса `10.244.0.3`.
* **Жизненный цикл:** Проводится мониторинг событий пода через `kubectl describe`. Это позволяет отслеживать этапы `Scheduled`, `Pulled` и `Started`.

### Блок 3 — Сложные конфигурации (YAML)
* **Декларативный подход:** Реализуется создание пода `my-webserver` через YAML-манифест.
* **Sidecar-паттерн:** В одном поде запускаются два контейнера: основной (`nginx`) и вспомогательный (`log-sidecar`), которые совместно используют дисковое пространство `emptyDir` для работы с логами.
* **Управление ресурсами:** В манифесте устанавливаются `limits` и `requests` для CPU и памяти, а также настраиваются пробы `liveness` и `readiness`.

## 2. Проблемы и способы их решения

| Проблема | Причина | Решение |
| :--- | :--- | :--- |
| **Ошибка `sh: kubectl: not found` внутри пода** | Попытка управления кластером изнутри контейнера приложения, где не установлен бинарный файл `kubectl`. | Выход из контейнера командой `exit` и выполнение команд управления на хостовой машине. |
| **Статус `ContainerCreating` дольше обычного** | Загрузка нескольких образов (`nginx:alpine` и `busybox`) для многоконтейнерного пода. | Ожидание завершения процесса через `kubectl get pods -w`. После скачивания слоев статус меняется на `Running`. |
| **Необходимость проверки Restart Policy** | Требуется убедиться, что Kubelet перезапустит упавший процесс. | Выполняется принудительное завершение основного процесса командой `kill 1`. Наблюдается автоматический перезапуск и инкремент счетчика `RESTARTS`. |

## 3. Ответы на контрольные вопросы

### Какие поды в kube-system всегда должны быть Running?
Для стабильной работы кластера в статусе `Running` должны находиться следующие компоненты:
1.  **kube-apiserver** — центральный узел управления API.
2.  **etcd** — база данных состояния кластера.
3.  **kube-scheduler** — планировщик размещения подов.
4.  **kube-controller-manager** — менеджер контроллеров.
5.  **coredns** — служба разрешения имен внутри кластера.
6.  **kube-proxy** — сетевой прокси на каждой ноде.

### Отличие Pod vs Container
**Под (Pod)** — это минимальная единица управления в Kubernetes, представляющая собой группу из одного или нескольких контейнеров, имеющих общие сетевые ресурсы и хранилища.  
**Контейнер (Container)** — это стандартная единица программного обеспечения, упаковывающая код и зависимости, которая запускается внутри пода как изолированный процесс.
