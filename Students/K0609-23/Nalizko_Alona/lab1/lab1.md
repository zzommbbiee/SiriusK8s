Отчёт по лабораторной работе: «Linux-основы контейнеризации»

В рамках выполнения лабораторной работы изучаются механизмы изоляции процессов в Linux: namespaces, cgroups и chroot. Цель работы – своими руками создать изолированное окружение без использования Docker и убедиться, что контейнеризация строится на этих трёх базовых механизмах ядра.

---

### 1. Namespaces – изоляция процессов

Для начала изучается, в каких пространствах имён находится текущая командная оболочка. Выполняется команда `ls -la /proc/$$/ns/`. В выводе видно, что у каждого типа изоляции есть свой числовой идентификатор. Например, PID namespace имеет номер 4026531836.

Далее с помощью утилиты `lsns` выводятся все активные пространства имён в системе. В списке видно, что сессия использует все основные типы: mnt, pid, net, ipc, uts, user, cgroup, time.

Для запуска процесса в новом PID namespace выполняется команда `sudo unshare --pid --fork --mount-proc /bin/bash`. Внутри нового окружения проверяется PID текущего процесса командой `echo "Мой PID: $$"` – результат 1, что означает, что этот процесс стал «корневым» в изолированной системе. Команда `ps aux` показывает только два процесса: сам bash и выполненную команду ps. Остальные процессы хоста не видны – это и есть изоляция пространства имён процессов.

**Контрольный вопрос: Почему после exit процессы хоста остались нетронутыми?**  
Потому что namespace был создан только для дочерней сессии. Как только сессия завершилась, происходит возврат в исходное окружение, и на хост это никак не влияет.

Для проверки сетевой изоляции выполняется команда `sudo unshare --net /bin/bash`, запускающая оболочку с изолированной сетью. Команда `ip link show` показывает только loopback, и тот находится в состоянии DOWN. Никаких физических или виртуальных интерфейсов внутри нового пространства нет – полная сетевая изоляция.

### 2. Cgroups – ограничение ресурсов

Для работы с cgroup v2 сначала находится точка монтирования командой `ls /sys/fs/cgroup/`. Затем создаётся собственная группа: `sudo mkdir /sys/fs/cgroup/mytest`.

Устанавливается лимит CPU: `echo "20000 100000" | sudo tee /sys/fs/cgroup/mytest/cpu.max`. Эта запись означает, что на каждые 100000 микросекунд процесс может использовать не больше 20000 микросекунд, то есть 20% одного ядра процессора.

Для тестирования ограничения сначала устанавливается утилита stress-ng через `sudo apt install stress-ng -y`, так как изначально она отсутствует в системе. Затем запускается стресс-тест: `stress-ng --cpu 2 --timeout 30s &`. Эта команда создаёт нагрузку на два ядра. После запуска PID процесса записывается в созданную cgroup: `echo $! | sudo tee /sys/fs/cgroup/mytest/cgroup.procs`.

До добавления процесса в cgroup в утилите `top` оба процесса stress-ng загружают процессор на 100% каждый. После добавления в cgroup нагрузка падает до 20% в сумме на два ядра – ограничение срабатывает.

Проверка статистики выполняется командой `cat /sys/fs/cgroup/mytest/cpu.stat`. Показатель `nr_throttled` равен 0, что означает, что процесс работал в рамках установленного лимита и не пытался его превысить.

**Контрольный вопрос: Что произойдёт, если лимит памяти превысить?**  
Если процесс попытается использовать больше памяти, чем разрешено в cgroup, сработает OOM-killer (Out-Of-Memory killer). Он завершит процесс-нарушитель, чтобы защитить систему от падения из-за нехватки памяти.

### 3. Chroot – смена корневой директории

Для создания минимального rootfs создаётся папка `/tmp/myroot` и внутри неё структура каталогов: bin, lib, lib64, proc, dev. Это будущий корень для изолированной среды.

С помощью команды `ldd /bin/bash` определяются библиотеки, необходимые для работы bash. Их все нужно скопировать в `/tmp/myroot/lib/` и `/tmp/myroot/lib64/`. ldd показывает список библиотек: linux-vdso.so, libiniparser.so.4, libtinfo.so.6, libc.so.6 и ld-linux-x86-64.so.2.


Сначала копируются только bash и ls, но при попытке войти в chroot возникает ошибка, что файл не найден. После копирования всех библиотек, указанных в ldd, внутри chroot команда `ls /` всё равно не работает. Тогда проверяются зависимости самого `ls` через `ldd /bin/ls` и копируются недостающие библиотеки для него. Только после этого команда `ls /` внутри chroot начинает работать.

Выполняется вход в chroot: `sudo chroot /tmp/myroot /bin/bash`. Эта команда «переносит корень» в созданную папку. Внутри команда `ls /` показывает только те папки, которые были скопированы: bin, dev, lib, lib64, proc. Никаких /home, /etc, /usr – только пустая чистая среда. Для проверки выполняется `echo "Я внутри chroot!"`, и сообщение успешно выводится. Выход из chroot – команда `exit`.

### Ошибки и сложности

**Проблема с unshare без sudo.** При попытке запустить unshare без прав root возникает ошибка Operation not permitted. Добавление sudo решает проблему – для создания пространств имён требуются привилегии.

**Стресс-тест stress-ng не был установлен.** Команда stress-ng --cpu 2 --timeout 30s выводит ошибку «команда не найдена». Установка через sudo apt install stress-ng -y решает эту проблему.

**Копирование библиотек для chroot.** Сначала копируются только bash и ls, но при входе в chroot возникает ошибка. Приходится внимательно посмотреть ldd /bin/bash и скопировать все библиотеки. Однако этого оказывается недостаточно – внутри chroot команда ls / всё равно не работает. Тогда проверяются зависимости самого ls через ldd /bin/ls и копируются недостающие библиотеки для него. После этого внутри chroot команда ls / наконец срабатывает. Без копирования библиотек для каждой команды она просто не запускается, так как динамическая линковка требует всех зависимостей.

---

### Вывод

В ходе лабораторной работы на практике изучаются три основных механизма контейнеризации в Linux: namespaces изолируют процессы, сеть и другие ресурсы; cgroups ограничивают потребление ресурсов (CPU, память); chroot меняет корневую файловую систему процесса. Docker представляет собой удобную обёртку над этими механизмами, автоматизирующую их настройку и управление.
