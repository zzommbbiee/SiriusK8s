Отчёт по лабораторной работе: «Kubernetes: установка кластера, первые поды»

В рамках выполнения лабораторной работы изучается запуск Kubernetes-кластера, его основные компоненты и управление подами. Цель работы – научиться проверять состояние кластера, создавать и управлять подами императивно и декларативно через YAML, а также понять механизмы самовосстановления.

1. Состояние кластера
Для начала проверяется состояние узлов кластера командой kubectl get nodes -o wide. Узел pc находится в статусе Ready и выполняет роль control-plane. Внутренний IP-адрес узла – 192.168.78.150, версия Kubernetes – v1.34.6+k3s1, операционная система – Ubuntu 25.10.

Команда kubectl describe node pc | head -50 выводит детальную информацию об узле: метки, аннотации, условия (MemoryPressure, DiskPressure, PIDPressure – все False), ёмкость ресурсов (4 CPU, 7557728Ki памяти, до 110 подов). Важно, что узел не имеет Taints, поэтому на него могут назначаться обычные поды.

Далее проверяются системные поды в пространстве имён kube-system командой kubectl get pods -n kube-system. Все основные компоненты работают: coredns, local-path-provisioner, metrics-server, traefik (входящий контроллер) и svclb-traefik. Под helm-install-traefik-crd и helm-install-traefik находятся в статусе Completed – это нормально, они выполнили установку и завершились.

Команда kubectl get componentstatuses (или kubectl get cs) показывает статус компонентов Control Plane: etcd, scheduler, controller-manager – все Healthy. Хотя API устарел, компоненты работают корректно.

Для просмотра API-ресурсов кластера выполняется kubectl api-resources | head -20. Вывод показывает доступные типы объектов: pods, services, deployments, configmaps и другие, а также их краткие имена (например, po для pods, svc для services).

Версия кластера проверяется командой kubectl version. Client Version и Server Version совпадают – v1.34.6 и v1.34.6+k3s1 соответственно.

Контрольный вопрос: Какие поды в kube-system всегда должны быть Running?
В kube-system всегда должны быть Running следующие поды: coredns (DNS-сервер кластера), kube-apiserver (API-сервер), kube-controller-manager (менеджер контроллеров), kube-scheduler (планировщик), kube-proxy (сетевая прокси), etcd (хранилище данных). В K3s также traefik и local-path-provisioner обычно находятся в Running.

2. Первый Pod
Императивно запускается под с nginx командой kubectl run nginx --image=nginx:alpine --port=80. Статус пода отслеживается через kubectl get pods -w. Сначала под находится в состоянии ContainerCreating, затем переходит в Running.

Команда kubectl get pods -o wide показывает, на каком узле запущен под (pc), его IP-адрес внутри кластера (10.42.0.9) и статус готовности.

Внутрь пода выполняется вход через kubectl exec -it nginx -- sh. Внутри проверяется:

hostname – возвращает имя пода (nginx)

cat /etc/hosts – показывает записи DNS, включая собственный IP

env | grep KUBE – переменные окружения от Kubernetes (KUBERNETES_SERVICE_HOST, KUBERNETES_SERVICE_PORT)

ps aux – виден только процесс nginx и оболочка, PID namespace изолирован

ip addr – показывается сетевой интерфейс eth0 с IP из кластерной сети (10.42.0.9)

Логи пода просматриваются командой kubectl logs nginx. Вывод содержит информацию о запуске nginx, прослушиваемых портах и старте рабочих процессов.

Команда kubectl describe pod nginx выводит подробную информацию о поде: его IP, образ, порты, условия готовности, события (Scheduled, Pulling, Pulled, Created, Started). Это позволяет понять, какие шаги были выполнены при запуске пода.

3. Pod через YAML
Создаётся файл pod.yaml с декларативным описанием пода. В спецификации указаны два контейнера: nginx с пробросом порта 80 и ресурсными ограничениями (requests: 32Mi памяти, 50m CPU; limits: 64Mi памяти, 100m CPU), а также log-sidecar на основе busybox, который каждые 5 секунд пишет текущую дату в лог-файл. Для sidecar используется emptyDir том для обмена данными между контейнерами.


Под применяется командой kubectl apply -f pod.yaml. Статус отслеживается через kubectl get pods -w. На начальном этапе под находится в ContainerCreating, затем могут возникать ошибки ErrImagePull или ImagePullBackOff, если образ не может быть загружен. После успешной загрузки под переходит в состояние Running с готовностью 2/2.

Имена контейнеров в поде проверяются через kubectl get pod my-webserver -o jsonpath='{.spec.containers[*].name}'. Вывод показывает nginx log-sidecar.

Логи sidecar-контейнера выводятся командой kubectl logs my-webserver -c log-sidecar. Содержимое лога показывает, что скрипт работает и каждые 5 секунд добавляет новую запись.

Вход в конкретный контейнер выполняется через kubectl exec -it my-webserver -c nginx -- sh. Внутри проверяется, что nginx работает и volume с логами доступен.

Команда kubectl get pod my-webserver -o yaml | head -60 показывает расширенную версию пода с добавленными Kubernetes defaults: значения ресурсов, переменные окружения, метки и аннотации.

4. Самовосстановление
Для проверки механизма самовосстановления выполняется попытка убить основной процесс nginx внутри контейнера: kubectl exec my-webserver -c nginx -- kill 1. В образе nginx:alpine процесс запущен от непривилегированного пользователя, и команда kill 1 завершается с ошибкой Permission denied. Это демонстрирует, что даже в контейнере нельзя убить процесс PID 1, если он запущен не от root.

Для тестирования восстановления используется другой подход – принудительное удаление контейнера на уровне containerd с помощью sudo k3s crictl rm -f $(sudo k3s crictl ps --name nginx -q). После этого команда kubectl get pods -w показывает, что под переходит в состояние Running с 1/2, затем снова становится 2/2. Счётчик RESTARTS увеличивается на 1, что подтверждает, что kubelet перезапустил упавший контейнер.

Контрольный вопрос: Почему Pod не удалился, а перезапустился? Кто за это отвечает?
Pod не удалился, потому что в спецификации пода не было указано, что он должен завершиться. kubelet на узле, на котором запущен под, отслеживает состояние контейнеров. Если контейнер падает (процесс завершился), kubelet перезапускает его в соответствии с политикой перезапуска (по умолчанию Always). Это обеспечивает самовосстановление приложений.

Ошибки и сложности
Проблема с kill 1 в контейнере. При попытке убить процесс nginx командой kill 1 возникла ошибка Permission denied. В образе nginx:alpine процесс запущен от непривилегированного пользователя, и PID 1 не может быть завершён обычным способом. Для тестирования самовосстановления пришлось использовать crictl rm -f для принудительного удаления контейнера на уровне containerd, после чего kubelet перезапустил его, и счётчик рестартов увеличился.

Предупреждение об устаревших componentstatuses. При выполнении kubectl get componentstatuses появилось предупреждение, что API устарел в версии v1.19+. Однако сами компоненты были в статусе Healthy, что подтверждает работоспособность кластера.

Проблема с ImagePullBackOff при создании пода через YAML. При первом запуске пода my-webserver один из контейнеров долго находился в состоянии ImagePullBackOff. Это происходило из-за проблем с загрузкой образа. После повторной попытки образ загрузился, и под перешёл в состояние Running.
