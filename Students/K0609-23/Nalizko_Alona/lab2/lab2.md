2 лаба:Отчёт по лабораторной работе: «Docker: образы, Dockerfile, запуск»

В рамках выполнения лабораторной работы изучается процесс создания Docker-образов, их оптимизация с помощью multistage сборки, а также управление ресурсами контейнеров. Цель работы – научиться писать эффективные Dockerfile, понимать архитектуру слоёв Union FS и правильно запускать контейнеры с ограничениями.

1. Первый Dockerfile
Для начала создаётся простая директория для проекта и в ней файлы приложения. В директории ~/docker-lab создаются три файла.

Файл app.py содержит простое Flask-приложение, которое возвращает имя хоста контейнера и версию приложения из переменной окружения, а также эндпоинт для проверки состояния.

Файл requirements.txt содержит единственную зависимость: flask==3.0.0.

Плохой Dockerfile (намеренно неоптимизированный) имеет следующую структуру: базовый образ python:3.12, рабочая директория /app, копирование всех файлов, установка зависимостей и запуск приложения.

После сборки образа командой docker build -t myapp:bad . и запуска контейнера проверяется его работа через curl localhost:5000. Размер образа оказывается около 1.12 ГБ.

Контрольный вопрос: Почему образ такой большой?
Образ получается большим, потому что используется полный образ python:3.12, который весит около 1 ГБ. В него входят все системные утилиты, компиляторы, библиотеки для разработки, которые не нужны для запуска приложения. Вместе с этим копируются все файлы из текущей директории, включая служебные файлы, которые тоже попадают в образ.

2. Multistage build
Для оптимизации размера образа используется multistage сборка. Процесс разделяется на два этапа.

Первый этап (builder) использует образ python:3.12-slim, который уже меньше полной версии. На этом этапе создаётся виртуальное окружение и устанавливаются зависимости.

Второй этап (final image) использует образ python:3.12-slim для обеспечения совместимости с builder. На него копируется всё виртуальное окружение целиком, а также само приложение. Также добавляется создание непривилегированного пользователя для безопасности.

Файл .dockerignore создаётся для исключения ненужных файлов из сборки: кэш Python, git-директория, файлы окружения, документация и файлы Dockerfile.

После сборки образа myapp:good его размер оказывается 135 МБ – это примерно в 8 раз меньше, чем myapp:bad. Разница хорошо видна при выполнении docker images myapp.

Запуск контейнера выполняется с ограничениями ресурсов: память 128 МБ, CPU – половина ядра, политика перезапуска unless-stopped. Проверка работы приложения через curl localhost:5001 показывает версию 2.0, что подтверждает корректную передачу переменной окружения. Команда docker stats app-good отображает установленные лимиты.

3. Dockerfile из методички не подошёл
Изначальный вариант хорошего Dockerfile использовал python:3.12-alpine в финальном образе и копировал зависимости из /root/.local. При сборке возникли ошибки: в Alpine Linux отсутствуют некоторые системные библиотеки, необходимые для работы некоторых Python-пакетов, к тому же команда adduser -D отличается от стандартной useradd.

В итоге использовал другой вариант: финальный образ на python:3.12-slim (совместим с builder), виртуальное окружение создаётся через python -m venv и копируется целиком. Сборка прошла успешно, размер образа остался небольшим (135 МБ).

4. Исследование образа
Для изучения слоёв образа используется команда docker history myapp:good, которая показывает все слои, их размеры и команды, создавшие эти слои. Сравнение с myapp:bad наглядно демонстрирует, почему один образ больше другого.

Команда docker inspect myapp:good | jq '.[0].RootFS' выводит информацию о слоях файловой системы образа. Видно, что каждый слой имеет свою хэш-сумму, а Union FS объединяет их в единую файловую систему.

5. Docker Hub
Для публикации образа выполняется вход в Docker Hub через docker login. После успешной аутентификации образ тегируется с именем пользователя и репозитория: docker tag myapp:good alonaka/flask-demo:v1.0. Затем образ публикуется командой docker push alonaka/flask-demo:v1.0.

В процессе публикации видно, что часть слоёв уже существует на Docker Hub (помечены как Mounted from library/python), что экономит время и место. После завершения публикации образ удаляется локально командой docker rmi, а затем скачивается обратно через docker pull и запускается на порту 5002 для проверки.


Ошибки и сложности
Проблема с правами доступа к Docker socket. При попытке выполнить docker build без прав root возникает ошибка permission denied. Проблема решается добавлением пользователя в группу docker или использованием sudo.

Конфликт имён контейнера. При повторном запуске контейнера с тем же именем возникает ошибка, что контейнер уже существует. Решение – удалить старый контейнер через docker rm или использовать другое имя.

Ошибка в app.py. При первом запуске контейнера app-bad он сразу падал. Проверка логов через docker logs app-bad показала ошибку NameError: name 'nano' is not defined. Оказалось, что в файле app.py случайно была добавлена строка с названием редактора. После исправления ошибки контейнер запустился корректно.

Несовместимость Alpine с некоторыми пакетами. При использовании python:3.12-alpine возникали ошибки сборки из-за отсутствия системных библиотек. Замена финального образа на python:3.12-slim решила проблему.

Результаты выполнения
docker images myapp – образ myapp:bad весит 1.12 ГБ, образ myapp:good – 135 МБ

docker history myapp:good – показана история слоёв образа

docker stats app-good – отображаются лимиты памяти 128 МБ и CPU 0.5
