1LABS

Сделал две команды, чтобы получить топ-10 IP по количеству запросов.
Первая — через awk. Она берёт первое поле (IP) из обычного access.log, дальше сортирует, считает количество через uniq -c, снова сортирует по убыванию и выводит топ-10. В результате на первом месте IP 10.17.208.74 с примерно 11 860 запросами, при этом в целом нагрузка по IP распределена довольно равномерно.
Вторая — через jq, но уже для JSON-лога. Логика та же самая, просто сначала достаётся поле remote_addr. Здесь значения меньше, потому что сам JSON-лог содержит меньше записей.

3LABS

Скрипт `03_disk_break.sh` создал большой файл, из-за которого диск начал заполняться. По идее он должен был забить всё пространство, но в моём случае занял около 6 ГБ — будем считать, что диск условно “заполнен”. Это подтвердилось через `df -h /`.
Чтобы не искать по всей системе, я сразу пошёл в каталог `/var/lib/linux_break_lab`, где и лежал этот файл. Нашёл его и удалил.
Однако после удаления место не освободилось. Это значит, что файл всё ещё удерживается процессом.
Далее выполнил: "sudo lsof | grep deleted"
В выводе как раз были процессы, которые держали удалённый файл. Их оказалось три — потому что скрипт запускался несколько раз. После этого завершил эти процессы, и только тогда место на диске освободилось.

Скрипт 04_binary_break.sh создал четыре “сломанных” файла в /opt/break_lab, и нужно было разобраться с каждым из них.
mystery_no_exec — у файла не было права на выполнение. Исправил через chmod +x, после этого он нормально запустился и начал работать.
mystery_bad_interpreter — через head посмотрел первую строку и увидел shebang #!/bin/no_such_interpreter_break_lab, то есть интерпретатор не существует. Исправил на #!/bin/bash, после этого файл начал выполняться и вывел never.
mystery_truncated_elf — проверил через file, оказалось, что у бинарника повреждена структура (нет заголовков ELF). Такой файл уже не починить, он просто битый.
not_a_binary — file показал, что это обычный ASCII-текст. Внутри была строка echo hello, то есть это вообще не бинарник, а скрипт без shebang.

5LABS

`journalctl` показал, что сервис запускается и сразу падает с кодом `exit 1`. Посмотрел сам скрипт через `cat` - в конце действительно стоял явный `exit 1`, из-за чего systemd считал выполнение неуспешным.
Исправил это через `sudo nano`, заменив `exit 1` на `exit 0`. После этого проверил через `systemctl status`: теперь отображается `code=exited, status=0/SUCCESS`, то есть сервис завершается корректно и больше не уходит в цикл перезапусков.
