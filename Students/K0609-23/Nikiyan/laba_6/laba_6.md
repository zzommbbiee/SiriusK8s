**Отчет по лабораторной работе №6: Kubernetes – ConfigMap, Secret, PersistentVolume**
1. Чему научился 

В ходе выполнения лабораторной работы были освоены механизмы управления конфигурацией и хранением данных в Kubernetes.

Создан ConfigMap app-config из литералов и nginx-conf из файла. Проверены три способа передачи конфигурации в под: через envFrom, через env с выбором конкретного ключа и через монтирование как файла в volumeMounts. Логи пода config-demo подтвердили, что конфигурация успешно загружена.

Создан Secret db-credentials из литералов. Просмотр его YAML показал, что данные хранятся в формате base64 (кодирование, не шифрование). Под secret-demo успешно использовал секреты как переменные окружения. Получены знания о том, что по умолчанию Secret не зашифрован в etcd, и для настоящей безопасности требуется EncryptionConfiguration или внешние системы (Vault).

Развёрнут PostgreSQL с использованием PersistentVolumeClaim (PVC) размером 1Gi. В базу добавлены тестовые данные. После удаления пода Deployment автоматически создал новый под, и данные сохранились – доказательство работы persistent storage.

2. Возникшие проблемы и способы их решения

В ходе выполнения работы проблем не возникло. Все объекты созданы корректно, поды запустились, данные сохранились.

3. Ответы на вопросы

Вопрос: Почему Secret по умолчанию небезопасен и что делать?
В стандартной конфигурации Kubernetes Secret хранится в etcd в формате base64, который легко декодируется. Это не шифрование, а лишь кодирование. Любой, имеющий доступ к etcd или API с соответствующими правами, может прочитать секреты.
Что делать: включить шифрование EncryptionConfiguration в API Server с использованием провайдера aescbc или aesgcm, ограничить доступ к etcd, использовать внешние системы управления секретами (HashiCorp Vault, AWS Secrets Manager и т.д.).
