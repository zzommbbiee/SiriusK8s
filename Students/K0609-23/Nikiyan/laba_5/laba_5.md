**Отчет по лабораторной работе №5: Kubernetes – Deployment, Service, Ingress**

1. Чему научился 
В ходе выполнения лабораторной работы были освоены ключевые механизмы управления приложениями в Kubernetes.

Создан Deployment с 3 репликами на основе образа nginxdemos/hello:plain-text. Проверено, что ReplicaSet автоматически создаётся и поддерживает заданное количество подов.

Выполнен rolling update до версии latest без даунтайма: в параллельном терминале через curl в цикле трафик не прерывался, поды обновлялись по одному (maxSurge=1, maxUnavailable=0). После обновления сделан rollout undo – успешный откат к предыдущей версии.

Создан Service типа NodePort на порту 30080, трафик равномерно распределялся между подами (round-robin). Дополнительно создан Service типа ClusterIP для проверки доступа внутри кластера.

Настроен Ingress-контроллер. Создан Ingress с правилами маршрутизации: webapp.local/ → сервис webapp-svc, webapp.local/api → отдельный сервис api-svc. После добавления записи в /etc/hosts проверена работа обоих эндпоинтов.

Сформулированы различия между типами Service: ClusterIP, NodePort, LoadBalancer.

3. Ответы на вопросы

Вопрос: В чём разница между ClusterIP, NodePort и LoadBalancer?

ClusterIP – стандартный тип сервиса в Kubernetes. Сервис получает внутренний IP-адрес, доступный только внутри кластера. Используется для взаимодействия между компонентами приложения (например, бэкенд общается с базой данных).

NodePort – расширяет ClusterIP: на каждой ноде кластера открывается определённый порт (по умолчанию 30000–32767), и трафик на этот порт перенаправляется на соответствующий сервис. Это позволяет получить доступ к приложению извне кластера, используя IP-адрес любой ноды и номер порта. Удобно для разработки и тестирования, но не рекомендуется для production из-за необходимости указывать порты и возможных конфликтов.

LoadBalancer – интеграция с облачным провайдером (AWS, GCP, Azure и др.). Провайдер автоматически создаёт внешний балансировщик нагрузки с публичным IP-адресом, который распределяет трафик между нодами. Это production-решение, обеспечивающее единую точку входа и масштабирование. В локальных средах (minikube, k3s) тип LoadBalancer может не получить внешний IP и оставаться в состоянии pending.
