# Отчет по лабораторной работе №6: Kubernetes: ConfigMap, Secret, PersistentVolume

## 1. Чему научился (Результаты работы)
В ходе выполнения лабораторной работы были освоены механизмы управления конфигурацией и постоянным хранилищем в Kubernetes, реализующие концепцию изоляции данных и настроек от кода приложения (согласно подходу 12-factor app).
* Изучен ресурс `ConfigMap`. Проверены три способа передачи конфигурации внутрь пода: загрузка всех ключей сразу через `envFrom`, точечная передача ключей через `env` и монтирование конфигурационного файла (например, `nginx.conf`) в виде тома (`volumeMounts`).
* Освоена работа с ресурсом `Secret` для хранения чувствительных данных (паролей, логинов). Подтверждено, что данные передаются в контейнер в виде переменных окружения.
* Настроено постоянное хранилище с помощью запроса `PersistentVolumeClaim` (PVC) для базы данных PostgreSQL. 
* Успешно проведено тестирование персистентности (сохранности) данных: после создания записей в БД под был принудительно удален, однако после его автоматического пересоздания (Deployment) данные остались целы и доступны для чтения.

## 2. Возникшие проблемы и способы их решения

* **Ошибка отсутствия имени пода при подстановке команд (Subshell Error):** При попытке выполнить команду создания таблиц внутри пода БД `sudo kubectl exec -it $(kubectl get pod -l app=postgres -o name) ...` возникала ошибка `permission denied` к файлу `k3s.yaml` и `pod, type/name must be specified`. 
  **Решение:** Анализ показал, что проблема крылась в конструкции подстановки оболочки `$(...)`. Команда внутри скобок выполнялась от имени обычного пользователя (без `sudo`), из-за чего падала из-за нехватки прав доступа к конфигурации кластера, возвращая пустую строку в основную команду `exec`. Проблема решена добавлением `sudo` внутрь подзапроса: `sudo kubectl exec -it $(sudo kubectl get pod -l app=postgres -o name) ...`.

## 3. Ответы на контрольные вопросы

**Вопрос 1: Почему Secret небезопасен по умолчанию и что делать?**
По умолчанию объект `Secret` в Kubernetes не использует криптографическое шифрование. Данные внутри него просто кодируются в формат `base64`. Это означает, что любой пользователь, имеющий права на чтение секретов в кластере или доступ к системной базе данных `etcd`, может легко раскодировать их с помощью стандартной утилиты `base64 -d`. 
Для обеспечения настоящей безопасности необходимо либо настраивать шифрование на уровне API-сервера (с использованием `EncryptionConfiguration` и алгоритмов вроде `aescbc/aesgcm`), либо интегрировать кластер с внешними защищенными хранилищами, такими как HashiCorp Vault или AWS Secrets Manager.
