Отчёт по лабораторной работе: «Docker: образы, Dockerfile, запуск»

Блок 1 — Первый Dockerfile (плохой образ)
Создали простое Flask-приложение с app.py и requirements.txt. Написали плохой Dockerfile, который использовал полный образ python:3.12 (весит ~1 ГБ) и копировал все зависимости подряд.

Собрали образ командой docker build -t myapp:bad -f bad.Dockerfile .. Сборка заняла около 14 секунд, так как pip устанавливал зависимости в полную версию Python.

После сборки docker images myapp показал:

myapp:bad — 1.62 GB (дисковое использование) / 416 MB (сжатый размер)

Контрольный вопрос: Почему образ такой большой?
Потому что базовый образ python:3.12 включает в себя компиляторы, заголовочные файлы, отладочные инструменты и весь набор библиотек, нужных для сборки пакетов. Плюс в образ попали все файлы из текущей директории (включая лишние), а слои не оптимизированы.

Блок 2 — Multistage build (хороший образ)
Написали хороший Dockerfile с многостадийной сборкой:

dockerfile
# Stage 1: builder
FROM python:3.12-slim AS builder
WORKDIR /build

RUN python -m venv /opt/venv
ENV PATH="/opt/venv/bin:$PATH"

COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

# Stage 2: final image
FROM python:3.12-slim
WORKDIR /app

RUN useradd -m appuser

COPY --from=builder /opt/venv /opt/venv

ENV PATH="/opt/venv/bin:$PATH"
ENV APP_VERSION=2.0

COPY --chown=appuser:appuser app.py .
USER appuser
EXPOSE 5000
CMD ["python", "app.py"]

Первый этап (builder): на python:3.12-slim создаём виртуальное окружение /opt/venv и устанавливаем зависимости через pip.

Второй этап (final): на чистом python:3.12-slim создаём пользователя appuser, копируем только готовое виртуальное окружение из builder, добавляем код и переключаемся на непривилегированного пользователя (небезопасно использовать привелегированного).

Добавили .dockerignore, чтобы не копировать временные файлы.

Собрали docker build -t myapp:good -f good.Dockerfile .. Сборка прошла успешно.

После сборки docker images myapp показал:

myapp:bad — 1.62 GB

myapp:good — 201 MB (дисковое использование) / 48.7 MB (сжатый размер)

Разница более чем в 8 раз. Это достигается за счёт:

- использования slim версии (вместо полной);

- отделения установки зависимостей от финального образа;

- удаления временных файлов из финального слоя.

Запуск с ограничениями ресурсов
Контейнер из хорошего образа запустили с параметрами:

docker run -d -p 5001:5000 --name app-good --memory="128m" --cpus="0.5" --restart=unless-stopped myapp:good
Проверили работу через curl http://localhost:5001 – получили ответ:

Hello from container! Host: f78a01d44460, Version: 2.0
Статус 200 OK подтвердил, что приложение работает.

Вывод docker stats app-good показал:

CPU % — 0.02% (при простое, но лимит 0.5 ядра установлен)

MEM USAGE / LIMIT — 25.24 MiB / 128 MiB (19.72%)

Лимиты применяются корректно, контейнер не может использовать больше ресурсов, чем разрешено.

Блок 3 — Исследование образа
docker history myapp:bad показал слои: сначала идут 3 слоя с общим весом ~1 ГБ (установка системных пакетов, компиляторов, зависимостей), затем копирование кода и установка Python-пакетов.
Видно, что в плохом образе много слоёв с установкой apt-get и большими временными файлами, которые не были удалены.

docker history myapp:good показал гораздо меньше слоёв, причём все слои маленькие. Финальный образ не содержит компиляторов и системных заголовков.

docker inspect myapp:good | jq '.[0].RootFS' вывел список слоёв (8 слоёв). Каждый слой соответствует одной команде в Dockerfile и хранит только изменения.

Ошибки и сложности
1. Ошибка при установке dive
При попытке установить dive через wget на Windows возникла проблема с параметром -q – PowerShell не распознал его. Пришлось использовать команду Invoke-WebRequest -Uri "https://.../dive_0.12.0_linux_amd64.deb" -OutFile "dive.deb". После этого файл скачался, но установить его в контейнере Docker Desktop не удалось – dpkg не был доступен. Для анализа слоёв использовали docker history и docker inspect.

2. Предупреждение Invoke-WebRequest при curl-запросе
При проверке curl http://localhost:5001 PowerShell выдал предупреждение о риске выполнения сценария. Выбрали вариант Y, чтобы продолжить. После этого запрос выполнился, статус 200 и текст ответа пришли корректно. В следующих запросах использовали curl с параметром -UseBasicParsing, чтобы избежать этого предупреждения.

3. Dockerfile из методички не подошёл
Изначальный вариант хорошего Dockerfile использовал python:3.12-alpine в финальном образе и копировал зависимости из /root/.local. При сборке возникли ошибки: в Alpine Linux отсутствуют некоторые системные библиотеки, необходимые для работы некоторых Python-пакетов, к тому же команда adduser -D отличается от стандартной useradd.
В итоге использовал другой вариант: финальный образ на python:3.12-slim (совместим с builder), виртуальное окружение создаётся через python -m venv и копируется целиком. Сборка прошла успешно, размер образа остался небольшим (201 MB).
