Отчёт по лабораторной работе: «Linux-основы контейнеризации»


1. Namespaces – изоляция процессов
Что делали:
Сначала посмотрели, в каких namespaces живёт текущая командная оболочка (команда ls -la /proc/$$/ns/). Вывод показал, что у каждого типа изоляции есть свой идентификатор – например, PID namespace с номером 4026531836.

Проверили через lsns:
Утилита lsns показала все активные пространства имён в системе. Видно, что моя сессия (PID 1780) использует все основные типы: mnt, pid, net, ipc, uts, user, cgroup, time.

Что значит «запустить в новом PID namespace»:
Команда sudo unshare --pid --fork --mount-proc /bin/bash создала новый процесс, у которого внутри свой собственный PID 1. Этот процесс считается родительским. Внутри нового окружения echo $$ показало 1 –  стал «корневым» процессом в этой изолированной системе. А ps aux показал только два процесса: сам bash и команду ps. Остальные процессы хоста не видны – полная изоляция.

Контрольный вопрос: Почему после exit процессы хоста остались нетронутыми?
Потому что я вышла из изолированного пространства, а не убила процессы всей системы. Namespace был создан только для этой дочерней сессии. Как только она завершилась, я вернулась в исходное окружение, и никакого влияния на хост не осталось.

Сетевой namespace:
sudo unshare --net /bin/bash запустил оболочку с изолированной сетью. Команда ip link show показала только loopback, и тот в состоянии DOWN. Никаких физических интерфейсов внутри нет – полная сетевая изоляция.

2. Cgroups – ограничение ресурсов
Что делали:
Сначала нашли точку монтирования cgroup v2: ls /sys/fs/cgroup/. Затем создали свою группу: sudo mkdir /sys/fs/cgroup/mytest. Для помещения нужных процессов и задания правил.

Установили лимит CPU:
echo "20000 100000" | sudo tee /sys/fs/cgroup/mytest/cpu.max – это значит: на каждые 100000 микросекунд процесс может использовать не больше 20000 микросекунд, то есть 20% одного ядра.

Нагрузили систему и поместили процесс в cgroup:
Запустили stress-ng --cpu 2 --timeout 30s – он создал нагрузку на два ядра. Потом записали его PID в cgroup.procs.
В top до ограничения оба процесса stress-ng ели 100% CPU каждый. После добавления в cgroup нагрузка упала до 20% в сумме на два ядра – ограничение сработало.

Проверили статистику:
cat /sys/fs/cgroup/mytest/cpu.stat показал, что лимит применялся: nr_throttled 0 – значит, процесс не пытался вылезти за рамки, работал ровно в рамках лимита.

Контрольный вопрос: Что произойдёт, если лимит памяти превысить?
Вступит в дело OOM-killer (Out-Of-Memory killer) – он просто убьёт процесс, который съел больше разрешённой памяти. Это защита, чтобы один процесс не обрушил всю систему.

3. Chroot – смена корневой директории
Что делали:
Создали папку /tmp/myroot и в ней структуру: bin, lib, lib64, proc, dev. Это будущий мини-корень для изолированной среды.

Скопировали bash и его зависимости:
Команда ldd /bin/bash показала, какие библиотеки нужны для работы bash. Их все нужно скопировать в /tmp/myroot/lib/ и /tmp/myroot/lib64/.
ldd выводит список библиотек: linux-vdso.so, libiniparser.so.4, libtinfo.so.6, libc.so.6 и ld-linux-x86-64.so.2.

Сам chroot:
sudo chroot /tmp/myroot /bin/bash – эта команда «переносит корень» в созданную папку. Внутри команда ls / показала только то, что мы туда положили: bin, dev, lib, lib64, proc. Никаких /home, /etc, /usr – только пустая чистая среда.
Там я запустил echo "Я внутри chroot!" и убедился, что всё работает. Выход из chroot – просто exit.

Ошибки и сложности
1. Проблема с unshare без sudo.
Пыталась запустить unshare без прав root, но получила ошибку Operation not permitted. Добавление sudo решило проблему – для создания пространств имён нужны привилегии.

2. Стресс-тест stress-ng не был установлен.
Сначала команда stress-ng --cpu 2 ... выдала ошибку «команда не найдена». Установила через sudo apt install stress-ng -y – после этого всё заработало.

3. Копирование библиотек для chroot.
Сначала просто скопировала bash и ls, но при попытке войти в chroot получила ошибку, что файл не найден. Пришлось внимательно посмотреть ldd /bin/bash и скопировать все библиотеки. Но оказалось, что этого мало – внутри chroot команда ls / всё равно не работала. Тогда проверила зависимости самого ls через ldd /bin/ls и скопировала недостающие библиотеки для него. После этого внутри chroot команда ls / наконец сработала и показала нужные папки. Без копирования библиотек для каждой команды она просто не запускается – динамическая линковка требует всех зависимостей.
