Отчет по работе №5: Kubernetes — Deployment, Service, Ingress
В этой лабораторной работе я изучала механизмы публикации приложений и управления их жизненным циклом без прерывания обслуживания (Zero Downtime). В качестве базовой сущности я использовала Deployment, который управляет наборами реплик (ReplicaSet). Я создала деплоймент webapp с тремя репликами. Основной задачей было протестировать механизм RollingUpdate. Благодаря параметру maxUnavailable: 0, Kubernetes гарантирует, что при обновлении версии образа старые поды не будут удалены до тех пор, пока новые не перейдут в статус готовности (readinessProbe). При попытке отследить обновление через скрипт с curl и grep возникла иллюзия обрыва связи: из-за смены формата ответа в новой версии latest утилита grep перестала находить нужную строку, однако прямые запросы подтвердили, что трафик не прерывался. После успешного обновления я также протестировала откат к предыдущей версии командой rollout undo, убедившись в наличии сохраненных ревизий в rollout history.

Для обеспечения сетевой доступности приложения я использовала объекты типа Service. В процессе создания NodePort сервиса возникла ошибка provided port is already allocated для порта 30080. Я решила эту проблему, убрав жесткую привязку к конкретному порту в манифесте, позволив Kubernetes самостоятельно выделить свободный порт из доступного диапазона. Отвечая на контрольный вопрос о типах сервисов:

ClusterIP предоставляет только внутренний IP-адрес, доступный исключительно внутри кластера (подходит для связи фронтенда с базой данных).

NodePort открывает статичный порт на IP-адресе каждой ноды кластера, позволяя обращаться к сервису извне.

LoadBalancer запрашивает внешний балансировщик нагрузки у облачного провайдера (AWS, GCP), который сам перенаправляет трафик на NodePort'ы кластера.

Заключительным этапом стала настройка маршрутизации трафика L7 через объект Ingress. Здесь я столкнулась с ошибкой 404 page not found при обращении к настроенным путям. Анализ показал, что в исходном ingress.yaml были прописаны аннотации и ingressClassName для контроллера Nginx (minikube addons enable ingress). Поскольку мой кластер работает на базе k3s, в нем по умолчанию используется Ingress-контроллер Traefik, поды которого находятся в пространстве имен kube-system (поэтому ingress-nginx оказался пустым). Я адаптировала манифест, удалив Nginx-специфичные аннотации, после чего Traefik успешно применил правила: запросы на корневой путь / стали маршрутизироваться в webapp-svc, а пути /api — в api-svc. Для локального тестирования я также добавила маршрут в /etc/hosts, привязав домен webapp.local напрямую к локальному IP-адресу 127.0.0.1, что позволило корректно симулировать внешние запросы к кластеру.
