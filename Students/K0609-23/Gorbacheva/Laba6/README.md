Отчет по работе №6: Kubernetes — ConfigMap, Secret, PersistentVolume
В этой работе я на разбирала концепцию отделения конфигурации и данных от кода приложения (согласно методологии 12-factor app). На первом этапе я работала с ConfigMap, создав конфигурации двумя способами: из набора литералов и напрямую из файла nginx.conf. В тестовом поде я проверила все три метода доставки этих настроек внутрь контейнера: массовая загрузка через envFrom, точечная передача конкретного ключа в переменную окружения (valueFrom) и монтирование целого конфигурационного файла в файловую систему через volumeMounts. Проверила логи пода, увидела, что все переменные и файл успешно пробросились.

Далее я изучила работу с конфиденциальными данными через объект Secret, создав учетные данные для базы данных. Отвечая на важный вопрос: по умолчанию Secrets в Kubernetes не являются безопасными. Если посмотреть манифест через kubectl get secret -o yaml, видно, что пароли не зашифрованы, а лишь закодированы в формат base64. Любой человек, имеющий доступ к API кластера или напрямую к базе данных etcd, может легко прочитать их командой base64 -d. Для реальной защиты (production-окружения) необходимо дополнительно настраивать EncryptionConfiguration на уровне API-сервера для шифрования данных в etcd, либо использовать внешние системы управления секретами, такие как HashiCorp Vault.

Основную часть работы я производила организацию постоянного хранилища (Stateful) для базы данных PostgreSQL с помощью PersistentVolumeClaim (PVC). На этом этапе возникло несколько характерных системных ошибок. Сначала PVC завис в статусе Pending. Причиной стал указанный в манифесте storageClassName: standard, который актуален для minikube, в то время как мой кластер работает на k3s и требует класс local-path. (Сначала не заметила комментарий в файле который дали, когда полезла разбираться что не так уже изменила...) При попытке исправить класс "на лету" через kubectl apply я получила ошибку spec is immutable (спецификация неизменяема) — Kubernetes запрещает менять параметры хранилища у уже созданного запроса. Проблема решилась полным удалением зависшего PVC (kubectl delete pvc) и повторным применением исправленного манифеста, после чего статус наконец сменился на Bound.

Вторая заминка произошла при попытке подключиться к базе для записи данных. Команда kubectl exec выдала ошибку container not found ("postgres"). Это произошло из-за того, что я попыталась выполнить команду в тот момент, когда объект Pod уже был создан в системе, но сам контейнер внутри него еще находился в статусе ContainerCreating (скачивался образ и монтировался диск). Дождавшись перехода в статус Running, я успешно подключилась к СУБД, создала таблицу и добавила тестовую запись "бэм бэм бэм бэм". 0-0

Для финальной проверки я удалила текущий под PostgreSQL. Благодаря контроллеру Deployment, кластер мгновенно создал новый под на замену упавшему и автоматически подключил к нему тот же самый PersistentVolume. Зайдя в новый под и выполнив SELECT, я убедилась, что тестовая запись осталась на месте. Это доказывает, что жизненный цикл диска в Kubernetes не зависит от эфемерной природы самих подов.
