Отчет по работе №2: Docker: образы, Dockerfile, запуск
На первом этапе я создала базовое веб-приложение и написала для него намеренно неоптимизированный Dockerfile (bad), использующий полновесный базовый образ python:3.12. Сборка этого образа наглядно ответила на контрольный вопрос: итоговый размер составил около 1 ГБ, так как вместе с интерпретатором в него попали системные компиляторы, кэш пакетного менеджера и инструменты ОС, которые абсолютно не нужны для работы готового приложения.

Для решения проблемы избыточного размера я применила подход multistage build (многоэтапная сборка) и создала файл .dockerignore, чтобы исключить попадание кэша и служебных файлов git в контекст сборки. В стадии builder на базе образа slim устанавливались зависимости, а во второй стадии на базе минималистичного alpine копировались только исходный код и готовые библиотеки. Сравнение вывода команды docker images показало ОЧЕНЬ сильное уменьшение итогового веса оптимизированного образа по сравнению с первой версией. Для понимания я исследовала структуру слоев полученного образа с помощью команды docker history и визуализатора dive, увидев, как каждая инструкция RUN или COPY в Dockerfile формирует отдельный слой.

Следующим этапом стал запуск оптимизированного контейнера с применением жестких лимитов ресурсов (--cpus="0.5", --memory="128m"). На этом шаге возникла проблема: утилита docker stats показывала нулевое потребление CPU и RAM, а контейнер постоянно перезапускался. Проверка логов выявила ошибку ModuleNotFoundError. Причиной стал конфликт прав доступа: в хорошем Dockerfile был создан безопасный пользователь appuser, однако зависимости Python из первой стадии скопировались в системную директорию /root/.local, читать которую appuser не имел права. Для исправления ошибки я удалила нерабочий контейнер командой docker rm app-good и запустила его заново с флагом -u root. После этого Flask-сервер успешно прочитал библиотеки, стартовал, а docker stats начал отображать корректные метрики в рамках заданных ограничений!!

Заключительным шагом стала публикация результатов работы на docker hub. Я авторизовалась в системе через docker login, промаркировала свой легковесный образ новым тегом с указанием моего имени пользователя и загрузила его на Docker Hub с помощью docker push. Для проверки работоспособности я удалила локальную копию образа и запустила контейнер заново, убедившись, что Docker автоматически скачивает нужные слои из облака и успешно разворачивает приложение с нуля. ._.

В итоге изучила процесс контейнеризации приложения на Python (Flask), принципы работы слоистой файловой системы Docker (Union FS) и методы оптимизации размера образов.
