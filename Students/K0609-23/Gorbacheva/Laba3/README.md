Отчет по работе №3: Docker: сети, volumes, docker-compose
В третьей работе я изучила принципы развертывания многоконтейнерных приложений с использованием Docker Compose, а также механизмы сетевого взаимодействия и сохранения данных. На первом этапе я ознакомилась с базовыми типами сетей в Docker: bridge (изолированная сеть на одном хосте, используемая по умолчанию), host (прямое использование сетевого стека хост-машины без изоляции) и overlay (сеть, объединяющая контейнеры на разных физических хостах в кластере). Создав пользовательскую сеть типа bridge, я увидела, что встроенный DNS-сервер Docker позволяет контейнерам успешно находить друг друга по именам. Затем я протестировала механизм постоянного хранения данных (volumes) на примере СУБД PostgreSQL. Запустив базу данных с примонтированным томом, я внесла тестовые записи, после чего намеренно удалила контейнер. Повторный запуск нового контейнера с подключением того же тома доказал, что данные успешно сохранились и не зависят от жизненного цикла самих контейнеров.

Основную часть работы я писала манифест docker-compose.yml для поднятия полноценного стека из трех сервисов: базы данных (PostgreSQL), бэкенда (Python/Flask) и фронтенда (Nginx). При первой попытке запуска стека командой docker compose up я столкнулась с ошибкой all predefined address pools have been fully subnetted. Причиной стало то, что в демоне Docker закончились свободные пулы IP-адресов для автоматического создания новых мостовых сетей. Для решения этой инфраструктурной проблемы я применила ручное управление диапазонами адресов (IPAM), явно прописав в конфигурационном файле изолированную подсеть 172.100.0.0/24 для дефолтной сети проекта. Это обошло системный конфликт и успешно создало сетевое окружение для стека.

После старта контейнеров возникла вторая проблема: сервис бэкенда не смог пройти проверку состояния (healthcheck), из-за чего весь стек не мог перейти в рабочее состояние, а логи выдавали ошибку Connection refused. Глубокий анализ показал, что проблема кроется в маршрутизации внутри образа Alpine Linux. Утилита wget, используемая для проверки, обращалась к адресу http://localhost:5000 и пыталась разрешить localhost через протокол IPv6 (адрес ::1). В то же время приложение Flask прослушивало подключения исключительно по IPv4 (127.0.0.1). Для стабилизации работы я скорректировала параметры healthcheck в файле docker-compose.yml, заменив неоднозначный localhost на жестко заданный IP-адрес 127.0.0.1.

Устранение этой особенности окружения позволило бэкенду успешно пройти проверку и сообщить статус healthy, после чего контроллер запустил зависимый от него фронтенд. Я проверила сквозную работоспособность системы с помощью curl: Nginx корректно принял запрос, проксировал его на Flask, который, в свою очередь, извлек данные из PostgreSQL и вернул JSON-ответ. В завершение я протестировала механизм горизонтального масштабирования, увеличив количество реплик бэкенда до трех с помощью команды docker compose up --scale backend=3, убедившись, что Docker автоматически распределяет нагрузку и управляет множественными экземплярами одного сервиса.
